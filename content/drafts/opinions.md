# 技术观点与方法论

**草稿——待摩斯终审,未经确认不得上线**

> 用途:数字分身语料 / 网站观点区 / 同行交流路径的谈资
> 信息来源:E:\Wiki 知识库(wiki/overview.md 及 concepts/ 各概念页)、摩斯全局工作规则、各项目 CLAUDE.md 实践

---

## 1. 第一性原理:从问题本质出发,不照搬惯例

所有决策回到三个问题:要解决什么?最直接的路径是什么?从零设计会怎么做?

实战例子:数字分身没有用 RAG/向量库。个人语料只有几万字,整理成几个 markdown 全量放进上下文、靠 prompt caching 控成本,就是最优解——省掉 embedding 管线和向量库运维,少一个故障点。等语料超过阈值再改检索也不迟。**问题规模不需要的技术,再流行也不上。**

## 2. 约束先行:没有规范的工作空间不动手

新项目第一步永远是写 CLAUDE.md(给 AI 的项目宪法),新目录先定结构约定:什么放哪、怎么命名、何时清理。需要调整规范时先改文档、再改实践,不反过来。

为什么?因为 AI 时代「执行」已经不稀缺,稀缺的是让执行不跑偏的边界。给 AI 一个有宪法的工作空间,它的产出质量和可预测性完全不同——这是花在杠杆点上的时间。

## 3. 人机分工:人负责判断,AI 负责执行,拍板不外包

- 人:spec、约束、验收标准、最终拍板
- AI:大规模执行、生成、初稿
- 铁律:AI 的产出不 review 不算完成;改完必须跑验证(test / lint / build);速度再快,错误传播的成本也由人承担

推论:不准备 review 产出,就不要启动自动化。「Build the loop, stay the engineer.」

## 4. 把重复 3 遍的事 AI 化——但先沉淀成规范再自动化

重复出现的工作,路径是:先手工做懂 → 写成 SOP/检查清单 → 变成 AI 可执行的 skill → 有验证条件后才升级为自动循环。直接跳到「全自动」的自动化大多死于没有验收标准。

## 5. 薄 Harness,厚 Skills:策略沉淀在文本层,不固化进架构

编排层(harness)只保留稳定职责:触发、路由、状态、权限、硬闸门。具体流程、策略、执行纪律沉淀到可版本化、可迭代的 markdown skills 里。

理由:模型在快速变化,把当前模型的缺陷固化成架构,等于给未来挖坑;而文本层的流程知识可以随模型升级持续重写,甚至让强模型编写、小模型执行。

## 6. 多 Agent 的瓶颈是治理,不是数量:制度本身就是最好的 prompt

多 agent 系统失控的主因从来不是 prompt 写得不够花,是架构治理和上下文治理缺失。有效的做法是先定协作制度:谁起草、谁审议、谁决断、谁分发、谁并行执行、谁汇总验收——再让 agent 各就各位。

摩斯自己的开发编排就是这么跑的:契约先行 → 多实现 agent 并行 → 独立评审 agent 把关 → 人做最终验收。

## 7. 知识先编译再查询:知识库是决策生产线,不是收藏夹

收藏从不等于拥有。摩斯的知识库流程是「入库 → 编译 → 连接 → 决策 → 体检 → 回填」:每份材料入库时就编译成三层——原始来源与延伸阅读 / 关键词与关键句 / 方法论,并强制回答「这篇材料改变了我的哪个决策」。

配套纪律:SCHEMA 规范先行、每周维护 SOP、定期 lint(查矛盾、查过时、查孤页)。知识库的本质是让个人经验对 AI legible——没被记录和结构化的经验,对 AI 等于不存在。

## 8. 事实洁癖(Fact Hygiene):引用是习惯,不是学术礼仪

知识库里每条非常识性事实必须带脚注:来源 + 定位符 + 原文引句(或标注 [synthesis])。给不出引用的论断,要么去找证据,要么弱化表述,要么删掉。

深度研究系统把同样的纪律做进了机器:关键论断没有证据支撑,报告不许出厂。AI 幻觉时代,可追溯性就是信任的成本价。

## 9. 诚实工程:能力状态必须显式声明

系统能力只有四种状态:已验收 / 实验性 / 已规划 / 明确不做。任何文档和汇报不得把实验性说成已验收——这条规则同样约束 AI 协作者。

作品集也遵守同一原则:每个系统标注真实状态(已上线内部/准备上线/开发中),量化数字附证据,暂无证据的标「统计口径整理中」。不装完工。

## 10. Vibe Coding 有边界:能上线 ≠ 该上线

快速原型阶段 vibe coding 极好用;但产品一旦涉及登录、UGC、上传、AI 聊天、个人信息或成本接口,验收门就必须从「能跑」扩展到安全基线:认证、权限、内容审核、成本熔断、隐私、应急开关。AI 把上线速度拉快了一个量级,也把「没想清楚就上线」的代价拉近了一个量级。

## 11. 用户体验是最高准则,高于技术偏好

后端可以很复杂,但用户触碰到的每一层必须丝滑。这里的「界面」包括 CLI、对话、错误提示、系统反馈。几条具体原则:为目标设计不为功能设计;不要让用户思考;系统承担复杂性;反馈要引导下一步行动(不说「连接失败」,说「正在重试,预计 5 秒恢复」)。

连限流都可以是体验设计:摩斯的数字分身触发每日额度后不弹冷冰冰的 429,而是进入「下班模式」人设文案——错误状态也是展品。

## 12. Token 换时间:预算是生产力基础设施

把 token 预算当基础设施支出,用机器时间换人类时间。同时配硬护栏:月预算熔断、每设备限额、话题围栏。烧 token 可以,失控不行。
