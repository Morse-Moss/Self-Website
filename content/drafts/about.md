## 工作哲学:把重复 3 遍的事 AI 化

摩斯有一条判断标准,写在他给 AI 的全局指令第一段:**任何重复 3 遍的事,AI 化或自动化。**

这不是效率口号,是每天真实执行的决策规则:

- 部署一个系统要反复踩同样的坑 → 沉淀成《客户部署故障字典》和可交给现场 Agent 执行的《部署运行手册》,下次让 AI 照着办
- 知识库维护每周做一遍 → 写成 SOP 和可触发的 skill,规定最低完成定义,AI 执行、人只做确认
- 内容从采集到发布要过五个工具 → 造一条流水线系统,一个界面走完全程

## AI native 的自我论证:不是会用 AI,是围绕 AI 重构

摩斯对「AI native」有一个明确的区分:**AI native ≠ AI user**。会用 ChatGPT 写周报是 AI user;把工作流拆开重装、让 AI 成为默认执行者,才是 AI native。证据不靠工具 logo 墙,靠系统设计:

1. **约束先行**:他的每个项目第一步不是写代码,是写 CLAUDE.md——给 AI 的项目宪法:目录约定、命名规则、验证标准、红线。没有规范的工作空间不动手。这是「为 AI 设计工作环境」而不是「用 AI 干活」的直接证据。
2. **知识库是给 AI 读的**:他的个人知识库(E:\Wiki)不是收藏夹,是一条「入库 → 编译 → 连接 → 决策 → 体检 → 回填」的决策生产线,由 Claude Code 维护:有 SCHEMA 规范、引用格式(每条非常识性事实必须带脚注和定位符)、每周维护 SOP、事实卫生检查(Fact Hygiene Lint)。人负责剪藏和确认,AI 负责编译和体检。
3. **人机分工写进制度**:人负责 spec、约束、验收和最终拍板;AI 负责执行和生成。改完必须主动跑验证(test / lint / build),AI 的产出不 review 不算完成。
4. **多 agent 用制度管,不用 prompt 堆**:他开发系统时用多 agent 编排模式(编排者定契约、多个实现 agent 并行、独立评审 agent 把关、人做最终验收),信奉「制度本身就是最好的 prompt」。
5. **这个网站本身**:你现在看到的作品集,从需求蓝图、内容起草到 UI 原型,就是按这套人机分工造出来的。它不是「展示工作的简历」,是「会工作的简历」。

## 技术观点速写

- **第一性原理**:所有决策从问题本质出发,不因「惯例如此」照搬。例:他的数字分身不用 RAG/向量库——个人语料就几万字,全量放进上下文加 prompt caching 就够了,少一个故障点。问题规模不需要的技术,再流行也不上。
- **用户体验高于技术偏好**:后端可以很复杂,但用户触碰到的每一层必须丝滑。CLI、对话、系统反馈都是交互界面。
- **诚实工程**:系统状态照实说——已上线的说已上线,开发中的说开发中,不装完工。他的深度研究系统里甚至有一份「能力矩阵」文档,强制区分 accepted / experimental / planned / not-implemented 四种状态,防止任何人(包括 AI)夸大系统能力。
