## 招聘官视角

### 1. 摩斯的技术栈是什么?

老本行是程序员[待摩斯补充:年限与早期技术背景]。当前高频使用:Python(FastAPI、SQLAlchemy)、TypeScript/React 全栈,加上 AI 工程这一层——多 agent 编排、prompt/上下文工程、模型网关(provider 无关架构)、agent 运行时设计、评估与治理。三套系统都是这个组合打出来的。

### 2. 「AI native」到底是什么意思?不就是会用 AI 工具吗?

不是。会用 AI 工具是 AI user;把工作流围绕 AI 重构才是 AI native。他的每个项目第一步是给 AI 写项目宪法(CLAUDE.md),知识库是按「AI 可读」设计的,开发流程是多 agent 编排 + 人做验收。区别在于:AI user 的产能上限是自己的时间,AI native 的产能上限是自己的判断力。

### 3. 有哪些拿得出手的项目?

### 5. 他能带团队吗?管理经验如何?

## 同行视角

### 15. 多 agent 编排用的什么框架?

核心观点比框架名重要:多 agent 的瓶颈是治理不是数量,制度本身就是最好的 prompt。他的开发编排模式是:编排者定契约 → 多实现 agent 并行 → 独立评审 agent 把关 → 人做最终验收。工具层主要基于 Claude Code 的 agent/skill 生态[待摩斯补充:是否公开更具体的工具链细节]。

### 16. 数字分身为什么不用 RAG?

第一性原理:个人语料就几万字,整理成几个 markdown 全量放上下文 + prompt caching,比 embedding 管线 + 向量库少一个故障点、少一套运维。超过 ~50k tokens 再改检索。技术选型跟着问题规模走,不跟着热度走。

### 17. 知识库怎么维护的?真的有用吗?

规则先行:SCHEMA 定义结构和引用格式(每条非常识性事实必须带脚注 + 定位符),每周维护 SOP,定期 lint 查矛盾和过时。维护者是 Claude Code,人只做剪藏和确认。有没有用的判断标准很朴素:每份材料入库时必须回答「它改变了我的哪个决策」,答不上来就不值得入库。

### 18. 深度研究系统和市面上的 Deep Research 有什么区别?

一句话:把「不敢信」当核心问题。关键论断没有证据支撑,报告不许出厂(硬闸门,不是提示词请求);全程事件留痕可回放;能力矩阵强制区分已验收/实验性/规划中。另外它在向 Agent OS 内核方向演进——报告只是内核上的第一个应用。

### 19. 提示词/系统架构公开吗?

分身的系统提示词直接公开——页面上就有「想看我的系统提示词?源码在这」。防御姿态不如开放姿态,顺便当 prompt 工程展品。各系统是否开源在逐个决定中[待摩斯补充:开源计划与 GitHub 链接]。

### 20. 一个人 + AI 的开发模式,真实瓶颈在哪?

代码不是瓶颈,AI 写得又快又多。真实瓶颈有三个:视觉品味打磨(高级感来自几十次微调,AI 写代码、人眼做裁判)、AI 产出的质量调教(工程一天,调教两周)、内容与叙事(AI 起草,人终审)。所以他把人的时间明确花在判断、审美和验收上——这也是「一个人 + AI 操作系统」里「一个人」不可替代的部分。
