# 数字生命摩斯作品集参考分析

> 日期：2026-07-13
> 目标：从真实网站提炼适用于数字生命摩斯的结构原则，不复制视觉皮肤。
> 项目事实来源：`docs/research/project-evidence-matrix-2026-07-13.md`。

## 同类参考

### 1. AIKing / KimYx

来源：`https://www.aiking.dev/` 与摩斯提供的当前页面截图。

- Steal：首屏直接给姓名/身份，不用抽象口号代替自我介绍。
- Steal：作品目录以卡片承载，项目名、短说明、标签和 GitHub/访问网页 CTA 有清晰分层。
- Steal：把“选择题认识我”做成参与式入口，建立比普通 About 更主动的了解路径。
- Avoid：矩阵雨、荧光描边和大量技术标签容易让多个 AI 作品集看起来相似。
- Avoid：作品卡文字过多时，真实项目画面退居背景，浏览效率下降。
- Translate：数字生命摩斯保留“问摩斯”，但升级成真实 RAG 对话；作品卡减少标签，只保留真实截图、名称、状态、一句话和可用 CTA。

### 2. Lee Robinson

来源：`https://leerob.com/`，2026-07-13 实页。

- Steal：开场只回答“我是谁、现在做什么、我关心什么”，身份信息非常具体。
- Steal：正文里的 Writing、Code、Reach out 都是自然语句中的真实链接，没有虚构功能按钮。
- Avoid：纯文本对需要看作品界面的招聘者不够直观，也不能承载本项目四套系统的视觉证据。
- Translate：首页身份文案采用 Lee 的具体性，但项目区必须用真实系统截图补足视觉证明。

### 3. Rauno Freiberg

来源：`https://rauno.me/`，2026-07-13 实页。

- Steal：首屏只有一句身份判断和一个强互动视觉，记忆点集中。
- Steal：作品、笔记、历史版本和联系方式是少量明确入口。
- Avoid：强交互画布会遮挡正文，不适合需要快速判断技术项目真实性的访问者。
- Translate：数字生命摩斯只保留一个品牌记忆点，例如 Morse 信号/生命体动效；真实项目证据不能被大面积抽象视觉遮挡。

## 跨界参考

### 4. Linear Customers

来源：`https://linear.app/customers`，2026-07-13 实页。

- Steal：卡片标题是具体故事判断，例如“为什么选择”“如何构建”，而不是功能名堆叠。
- Steal：同一目录区分 Read story、Watch video、Visit site，不强迫所有项目进入同一种详情路径。
- Avoid：大量客户和分类适合企业目录，但四个项目不需要复杂筛选。
- Translate：四个项目不加筛选栏；每张卡按真实状态显示“查看案例”“访问系统”或“GitHub”。

### 5. Pentagram Work

来源：`https://www.pentagram.com/work`，2026-07-13 实页。

- Steal：项目本身是第一视觉，标题和一句说明为图片服务。
- Steal：目录与详情分层，首页不承担完整案例解释。
- Avoid：过多留白与懒加载空白会降低首屏信息效率；极简目录也不适合没有高质量图片的项目。
- Translate：作品目录采用大图和不对称节奏，但只有真实截图可进入图像位；缺图项目显示克制的“截图待补”，不用生成 UI。

### 6. Stripe Customers

来源：`https://stripe.com/customers`，2026-07-13 实页。

- Steal：案例用“挑战 -> 解决方案 -> 产品/证据”组织，访客能快速理解为什么这样设计。
- Steal：案例列表以多种真实视觉并置，而不是所有卡片套同一张装饰图。
- Avoid：企业级大数字、Logo 墙和营销成果不适用于当前没有公开业务数据的个人作品集。
- Translate：案例详情采用“问题、关键判断、系统结构、验证证据、当前边界”；不设置成果大数字区。

## 转译后的设计原则

1. 首页 H1 固定为“数字生命摩斯”，副标题说明本人是 Agent 系统开发者；抽象理念只能进入支持文案。
2. 第一屏必须同时让访客看到本人身份和至少一个真实系统画面，且下一段作品内容在首屏底部露出。
3. 首页项目区只展示四个系统：内容创作 Agent 系统、自动运营 Agent 系统、深度研究 Agent 系统、数字摩斯。
4. 项目卡采用真实截图优先；文字控制为状态、名称、一句话和最多两个操作。
5. CTA 根据事实变化：自动运营 Agent 系统与未来部署项目显示“访问系统”；深度研究 Agent 系统与数字摩斯显示 GitHub；内部系统无公开地址时不造按钮。
6. 需要解释的项目进入案例详情，统一采用“问题 -> 我的角色 -> 关键判断 -> 真实结构 -> 验证证据 -> 当前边界”。
7. “问摩斯”是认识本人的交互入口，不是虚假的数字人表演。数字人形象未完成前使用对话 UI，不放生成角色图。
8. 首页不放业务效果数字。测试、构建、评测和真实联调只进入案例证据区。
9. 视觉记忆点来自 Morse 信号、信息轨迹和真实系统图的编排，不复制矩阵雨、赛博边框或通用 AI 光球。
10. 项目证据与概念素材分库；`docs/verify/concepts/imagegen-*.png` 永远不进入作品证据位。

## 明确禁区

- 不用一整行抽象大字代替姓名和身份。
- 不在首页堆长段项目说明、技术栈标签或测试数字。
- 不把生成图、CSS 流程图、蓝图或历史截图伪装成当前真实界面。
- 不把“已实现但实验性”写成成熟产品能力。
- 不制造 GitHub、访问系统、联系方式或数字人入口。
- 不复制 AIKing 的矩阵雨和荧光标签皮肤。
- 不使用公司、客户、账号、业务数据、内部地址和源码。
