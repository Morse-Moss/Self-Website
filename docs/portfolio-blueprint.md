## 1. 定位与核心叙事

- **网站性质**:不是「展示工作的简历」,是「会工作的简历」——网站本身就是 AI native 能力的第一件展品
- **一句话定位**:数字生命摩斯——一个人 + 一套 AI 操作系统
- **首要诉求**:突出个人稀缺性,AI native 优先(高于大厂投递、接客户等具体场景)
- **核心论证逻辑**:AI native ≠ 会用 AI 工具(AI user),而是工作流围绕 AI 重构。证明方式靠系统设计,不靠工具 logo 墙
- **叙事主线**:「把重复 3 遍的事 AI 化」的工作哲学 → 每个系统讲「痛点 → 人机分工 → 量化收益」的杠杆叙事
- **现实约束**:各系统尚未开发完/未上线 → 展厅走 **Building in Public** 风格,不装完工,展示进度,系统上线一个点亮一个

## 2. 受众与双速设计(已确认)

大厂招聘官没耐心 → 给两种阅读速度各修一条路:

- **30 秒速览层**:首屏不滚动、不等动画即可见:名字、一句话定位、核心技术栈、3 个硬亮点、联系方式。特效永远不挡信息
- **私密简历入口**:右上角保留简历入口；访客通过一次性邀请兑换独立 Session 后，才能在浏览器顶层打开同源加密 PDF。公开页面不嵌入结构化简历，也不以浏览器存储或 DOM class 作为授权依据
- 分身、特效、深度内容全部自愿深入,无强制路径

## 3. 信息架构(一期范围)

```
首页(深蓝高级感 + 数字生命形体主视觉 + 30秒速览层)
├── 系统展厅(Building in Public,三层深度)
├── 数字分身(全站常驻,核心功能)
├── 统计账单页(倾向一期做简版,待最终确认)
└── 私密简历访问(邀请授权弹窗 + 同源 PDF)
```

语言:中文主,英文二期。技术栈:Next.js + Vercel(分身需要服务端 API;将来可原样迁自有服务器)。

### 3.1 系统展厅:三层深度

- **L1 介绍卡**:痛点 → 方案 → 人机怎么分工造的 → 当前状态(已上线/开发中 + 进度)
- **L2 一键直达**:已上线系统新窗口直开,链接带来源标记(统计作品集导流数据,未来是账单页素材)
- **L3 站内试驾间**:深度研究系统嵌入作品集直接体验(见 3.2)

### 3.2 试驾间(深度研究系统站内调用)

- **示例回放为主(已确认)**:预跑 3-5 个真实研究案例,访客按时间线查看 agent 状态、工具事件、搜索摘要、公开工件与评审结果;不展示隐藏思维、原始提示词或原始模型输出。零 API 成本、零翻车风险,震撼力约为真实调用九成
- **限额真实调用:一期不做(已定案)**,二期再加。届时「今日试驾名额剩 N 个」全站限额,用完降级回放;结果可导出/发邮箱(顺带留资线索);护栏:输入过滤、单次预算熔断、排队机制

### 3.3 数字分身(核心功能)

- **人设:方向 A「数字生命本体」**——分身就叫数字摩斯,真人摩斯造出的数字生命,第一人称说话。品牌闭环:ID 即产品。示例开场:「我是数字摩斯,真人摩斯造的数字生命。他负责睡觉和写代码,我负责 24 小时值班。」
- **开场分流**:分身先说话,不等访客开口。三个按钮:招聘 / 合作 / 同行交流,各进定制路径(解决 chatbot 冷启动死穴)
- **招牌能力一:JD 匹配报告(已确认方向)**:招聘方贴 JD → 现场生成《摩斯 × 该岗位》匹配分析,含对口项目、能力证据、诚实的短板
- **招牌能力二:需求初诊(已确认方向)**:客户描述问题 → 初步思路框架 + 预约真人 30 分钟。分身替真人做售前筛选
- **可信度设计**:会认怂——没做过的明确说没做过,给相邻经验;回答尽量带证据链接(指向具体系统/代码)
- **公开 Prompt 当展品**:不防套系统提示词,页面直接放「想看我的系统提示词?源码在这」,防御姿态变开放姿态,顺便秀 prompt 工程
- **知识来源**:现有 Wiki 知识库(**E:\Wiki**,持续更新中)+ 各系统项目资料,提炼「公开版知识库」;涉隐私内容过人工审核。私密简历是独立的人工终审 PDF，不进入公开知识库、RAG、Provider 输入或分身语料
- 工程要点:防滥用(免费额度、rate limit、话题围栏只聊摩斯相关)

### 3.4 分身技术方案(2026-07-07 定案)

**架构**(一切以「一人 + AI 可落地」为准绳):

```
浏览器(SSE 流式渲染)
  → Next.js API Route /api/chat
    → 限流层(IP + 设备指纹,Upstash Redis 免费档)
    → 系统提示词(人设 + 规则)+ 公开版知识库(全量入上下文,prompt caching)
    → Claude API(流式)
```

- **知识库不上 RAG/向量库**:个人语料仅几万字,整理成 5-6 个 markdown(about / systems/* / opinions / collaboration / faq)全量放进上下文,靠 prompt caching 控成本。省掉 embedding 管线和向量库运维,少一个故障点;超过 ~50k tokens 再改检索。「我的分身为什么不用 RAG」本身可写成网站上的架构决策故事(第一性原理:问题规模不需要)
- **模型:摩斯自选(已定案:架构做成 provider 无关)**。通过 Vercel AI SDK 的 provider 抽象接入,模型/厂商是配置项(环境变量),换模型不动代码。选型时关注三点:流式输出支持、长上下文(知识库全量注入需要)、国内调用可达性。可配置双模型:日常对话用便宜的,JD 报告等高价值任务用强的
- **费用**:按所选模型重估(国产模型普遍比 Claude 便宜一个量级);无论选谁,**月预算硬熔断机制保留**
- **防滥用**:每设备每日 20 条、单条 500 字、话题围栏(只聊摩斯相关);触发限制时进入「下班模式」人设文案(「今天聊得够多了,数字生命也要充电,明天再来」)——把限流做成人设的一部分,错误状态也是展品
- **UI**:首屏点击形体 → 形体退到对话区一侧,对话在主区展开(保持「在和它说话」);其他页面右下角常驻小形体;回答中提到系统时渲染成可点卡片,引流到展厅/试驾间;移动端全屏抽屉
- **对话日志要存**(脱敏):迭代燃料(每周看答不上来的问题补语料)+ 访客情报(谁在了解你)

**知识库生产流程**:① 只收集获准公开的知识库导出与项目文档，不接收私密简历 → ② AI 按模板起草公开版 → ③ 摩斯终审删隐私 → ④ 上线后每周依据对话日志补料。人只做 ① 的收集和 ③ 的终审。

### 3.6 数字人口播方案(2026-07-08)

- **形象来源:实拍素材克隆**——摩斯录 2-3 分钟真人视频训练专属数字人形象;AI 生成写实形象与风格化形象两条路线否决
- **一期**:①预渲染口播自我介绍(平台离线生成 mp4,网站 `<video>` 点击播放,带字幕);②分身语音回答音频版(LLM 流式文本 → 克隆声 TTS 流式,数字人形象播「说话循环」动画,不做逐字唇形)
- **二期**:「带我逛」全站导览(分段解说词 + 滚动编排 + 预生成音频,随时退出)
- **否决**:实时唇形同步视频(HeyGen Interactive / D-ID 类,按分钟计费,公开站成本不可控);进站自动播放带声口播(浏览器 autoplay 政策禁止带声自动播,且违反双速设计)
- **平台候选(待拍板)**:形象克隆——可灵(单条质量评测口碑最好,积分制按条付费,适合低频高质量)或腾讯智影(99-199 元/月,量产便宜但表现力最弱);声音克隆——推荐火山引擎豆包(150 元/年/音色,流式 <300ms,中文第一梯队),备选 Fish Audio(按量最便宜,开放权重)
- **TTS 同样走 provider 无关**:声音供应商为配置项;月预算熔断覆盖 TTS 调用;所有克隆需本人授权验证(各平台合规要求)

## 4. 视觉方向(已拍板:形体概念成立,L1 工艺目标 + L0 降级)

- **色彩**:深蓝偏好(已确认)。深空蓝基底(约 #0A0E1A 一带,非纯黑)+ 青色系辉光
- **策略(用户原话)**:可以结合主流,但不要只追随主流;要更有设计感、艺术感
- **核心概念:数字生命形体**——首屏中央一团深蓝色生成式流体/光体(WebGL),缓慢呼吸、随鼠标流动、每次访问形态不同。**它就是数字摩斯的身体**:点它开始对话,说话时泛波纹。特效=主角、分身=有形象、ID=字面呈现,三合一;别人抄不走(抄了不成立)
- **组合公式**:lusion 的工艺克制 + 生成式流体「活体」概念 + 编辑级排版信息层,统一深蓝色域
- 参照:[lusion.co](https://lusion.co)(工艺上限校准)、[piellardj.github.io](https://piellardj.github.io/)(流体质感)、[Pangram Pangram](https://pangrampangram.com) 与 [Tom Sears](https://www.awwwards.com/tom-sears-personal-portfolio.html)(编辑排版/印刷品思维)、[Brittany Chiang](https://brittanychiang.com/)(经典深藏青开发者站)、[Linear](https://linear.app)(主流高级感基准)

### 4.1 数字生命形体·可行性分级(一人 + AI 落地)

| 级别 | 内容 | 实现 | 评估 |
|---|---|---|---|
| L0 保底 | 动态深蓝 mesh gradient(Stripe 风渐变动画)+ 微噪点 | 纯 CSS/Canvas,有开源实现 | ~1 天,已具高级感;移动端/低性能设备的降级形态 |
| L1 目标(推荐) | Three.js 噪声球体:icosahedron + simplex noise 顶点位移 + fresnel 辉光,鼠标交互 + 对话呼吸态 | react-three-fiber + drei 生态,社区经典配方,教程/开源遍地,AI 生成可靠 | 2-4 天含调优,95% 访客的「哇」阈值 |
| L2 进阶(不追) | 真流体模拟/自定义 shader 交互(lusion 级) | 专业工作室门面水准 | 二期都不一定做;差异化在概念不在工艺极限 |

原则:渐进增强——设计保证 L0 也成立,检测设备性能自动降级。

## 10. M3-RAG MVP 增量决策(2026-07-12)

- 产品主体仍是作品集网站,不是独立客服 SaaS。招聘方/面试官为第一受众,客户/合作方第二,内容读者第三。
- 首页、项目、关于和公开内容无需登录;短期邀请码只解锁数字摩斯实时文字对话与面试官模式。
- 回答知识只来自网站公开内容和摩斯审核通过的扩展知识;本阶段禁止读取 `content/drafts/` 和外部本地仓库。
- 本阶段直接加入真实 RAG:PostgreSQL + pgvector,不单独部署 Milvus/Qdrant;完成切片、Embedding、向量召回、来源展示和检索验证。当前是受控访问、低并发、小规模公开知识库,pgvector 能让向量、内容、会话和用量共用一套事务、备份与访问控制,少维护一个网络服务和一套权限。独立向量库保留为扩展选项,只有基准测试证明 PostgreSQL 的检索延迟或写入吞吐不达标,或出现需要独立扩缩容、多租户强隔离、专用混合检索能力时再迁移;迁移必须通过现有 provider/retrieval 边界完成,不能让业务层绑定 Milvus/Qdrant。
- 首发使用 OpenAI GPT,但模型 ID、Embedding 模型和 base URL 全部配置化;不得把 ChatGPT/Codex 订阅当成 API 额度。
- 邀请码有效期内保留短期对话;到期后清理身份关联。长期知识只能由对话生成候选,经摩斯审核后进入知识库。
- 成本治理采用单邀请码额度 + 全站月度预算;50/75/90/100% 分级,100% 时普通对话熔断并保留静态作品集。
- 联网搜索、语音、TTS、实时数字人口型和长期访客画像暂不进入首次跑通阶段。

## 15. S10 数字摩斯智能客服(2026-07-15)

> 本节是 S10 及后续实现的有效产品决策，采用后决策覆盖原则。§3.4、§5、§10、§12 和 §14 中关于“无 RAG 全量上下文”“月预算硬熔断”“日志脱敏”“首阶段不联网”和“S8 不含管理后台/通知”的表述保留为当时历史，不再约束 S10。

- **产品目标**：把现有受控文字对话补成可本地验收的智能客服。三种流程为自由对话、JD 匹配和需求初诊；不是独立客服 SaaS。
- **访问与记忆**：邀请码最长 72 小时；每次 access session 和可恢复上下文为 12 小时，不做跨 Session 长期记忆。
- **RAG**：使用本地 BGE + PostgreSQL/pgvector，公开知识唯一 live source 继续是 `content/site-content.json`；不部署 Milvus/Qdrant，不读取 `content/drafts/**` 补答案。
- **模型**：使用配置化 OpenAI-compatible 中转 GPT；协议明确选择 Responses 或 Chat Completions，单次请求不自动跨协议或跨模型回退。模型、主备 endpoint 顺序、reasoning effort、超时和并发均由服务端配置；同模型、同协议的备用 endpoint 只允许在尚未输出正文时接管。
- **自动联网**：由服务端判断是否调用博查，访客没有搜索开关。每轮最多一次、每 Session 最多五次；S10 只消费标题、摘要和经校验的 HTTPS URL，不抓任意网页正文，不把搜索结果写回知识库。
- **事实与引用**：Morse 的履历、项目状态、数字、联系方式和能力事实只能由站内审核知识确认；网页只补外部背景。引用链接只能由服务端产生，模型不能自行生成可点击 URL。
- **数据保留**：运行态 history 随 12 小时 Session 清理；原始问题、回答、搜索词和来源另存 10 天，不脱敏，仅私有管理后台可见。页面不向访客展示数据保留提示；10 天后删除原文，只保留无正文聚合。
- **成本与防滥用**：取消月预算硬熔断；保留每 Session 30 条消息、五次联网、独立并发/超时/kill switch、Token/估算费用统计、邀请码与管理员登录限流。usage 缺失记未知，不伪造为 0。
- **飞书**：只通知首次邀请码使用、需求初诊、服务故障/恢复和安全攻击。业务事务写 Outbox，提交后再发送；失败不回滚访客回答，稳定事件 key 保证同一业务事件不重复入队。普通 webhook 投递采用至少一次语义；若飞书已接收但本地 `sent` 落库结果未知，极端情况下可能重复，严格恰好一次需要后续接入支持服务端幂等键的投递接口或中介。普通聊天、JD 和常规额度不通知。
- **管理后台**：`/admin` 不进入公共导航；使用独立管理员密码认证，支持 10 天记录筛选、详情、badcase、JSON/CSV 导出，导出前重新验证管理员密码。管理员可在同一私有控制台生成、查看状态和停用短期邀请码；创建必须持有有效管理员 Session 并校验精确 Origin，由服务端生成 `morse_` 前缀随机码，数据库只保存 SHA-256。明文只在创建成功时显示一次，关闭后不可恢复；停用只阻止新兑换，不注销已经建立的访客 Session。管理员与访客 Session 物理隔离，管理员登录保留五次失败锁定、30 分钟滑动过期和 HttpOnly/Secure/SameSite=Strict cookie。
- **UI**：保留 S9 对话外观，只增加三个 workflow、服务端阶段状态、真实停止、12 小时历史恢复和站内/联网来源分组；没有独立页面的当前站点资料显示为静态依据，站内项目案例与联网资料均在新标签页打开，不能替换或打乱当前对话；不重设计作品集。
- **本轮非目标**：语音、TTS、数字人视频、口型、跨 Session 长期记忆、自动知识发布、网页正文抓取、外置向量库、部署和 push。
