# 私密简历访问与加密存储设计

日期：2026-07-20
状态：设计已确认，待实施计划
预期执行合同：`STAGED / CRITICAL / DEPLOYED`

## 1. 目标

在公开作品集保留“简历模式”入口，但把真实简历改造成独立的私密资源。只有获得一人一码授权的熟人可以在限定时间内查看、打印或保存最终 PDF；简历正文和个人隐私不得进入公开页面、公开知识库、对话模型或普通聊天授权范围。

本设计保护的内容包括但不限于真实姓名、电话、邮箱、微信、城市、公司、教育经历、工作经历以及简历中的其他个人信息。

## 2. 当前实现与风险

当前 `ResumeModeToggle` 只通过 `localStorage` 和 CSS class 切换显示状态，`ResumeSheet` 始终由服务端渲染并随页面返回。该机制是显示偏好，不是访问控制。

当前公开知识摄取会从 `content/site-content.json` 的 `profile` 提取资料。若把真实简历或联系方式写入现有 profile、`ResumeSheet` 或公开内容文件，信息可能出现在 HTML、RSC 数据、前端构建产物、向量库或聊天上下文中。

因此，真实简历不得写入：

- `content/site-content.json`、`content/drafts/` 或其他 Git 跟踪文件；
- `public/`、Next.js 静态资源或构建镜像；
- 公开知识文档、Embedding 输入、RAG chunk 或评测夹具；
- 对话消息、Provider 请求、应用日志、截图或验收证据。

## 3. 已确认的产品规则

- 聊天权限与简历权限完全分离。
- 普通聊天邀请码只能解锁对话，不能访问简历。
- 简历邀请码只能解锁简历，不能升级聊天权限。
- 简历邀请码一人一码，由管理员填写熟人备注。
- 邀请码创建后 7 天内可兑换，只能成功兑换一次。
- 兑换成功后，邀请码立即失效；同一浏览器获得 72 小时简历 Session。
- 公开页面继续显示“简历模式”入口，未授权时只显示独立的邀请码验证界面。
- 授权用户可以在线查看、打印或保存干净的最终 PDF，不添加水印。
- 系统不提供 DOCX 等源编辑文件，也不生成公开下载地址。
- 系统只保留一份当前简历。上传新版后，已有有效 Session 继续生效并自动看到新版。
- 后台停用邀请码时，由该邀请码产生的全部简历 Session 立即失效。
- 原始 IP、User-Agent 和设备信息用于访问审计，保留 30 天后自动删除。
- 系统只能准确记录 PDF 成功返回，不能声称识别了浏览器中的真实打印或保存动作。

## 4. 安全不变量

以下规则必须由服务端和自动化测试共同保证：

1. 未授权请求永远拿不到 PDF 明文、个人信息或可复用文件地址。
2. 页面是否显示简历正文只能由有效的服务端简历 Session 决定，不能由 CSS、`localStorage` 或客户端参数决定。
3. 简历 PDF、简历 Session 和简历邀请码与聊天系统使用不同的表、Cookie、接口和权限检查。
4. 即使请求同时携带聊天 Session 和简历 Session，聊天服务也没有读取、检索或附加简历内容的代码路径。
5. 简历正文永远不进入公开 RAG、Search、Embedding、模型输入或对话历史。
6. 加密密钥不进入代码、数据库、管理后台、日志、错误响应或构建镜像。
7. 解密、鉴权或完整性校验失败时必须拒绝返回，不能降级读取明文副本。
8. 停用授权后，下一次文件请求必须立即失败；已打开页面刷新后不能继续读取。

## 5. 总体架构

### 5.1 公开入口

公开页面保留“简历模式”按钮。按钮不再直接切换 `resume-mode` class，而是先请求简历授权状态：

- 未授权：打开简历邀请码验证界面；
- 已授权且未过期：进入私密 PDF 查看界面；
- 没有已上传简历：显示“简历暂不可用”，不返回占位简历或历史文件。

现有结构化 `ResumeSheet` 和基于 `localStorage` 的预加载授权判断应移除。客户端可以保存非敏感的界面偏好，但不能保存授权结论或简历正文。

### 5.2 独立授权域

新增独立的简历访问域：

- `POST /api/resume/access`：兑换简历邀请码；
- `GET /api/resume/access`：读取当前简历授权状态；
- `DELETE /api/resume/access`：退出当前简历 Session；
- `GET /api/resume/file`：鉴权后解密并返回当前 PDF。

简历 Session 使用独立 Cookie，例如 `morse_resume_access`，属性必须为 `HttpOnly`、`Secure`、`SameSite=Strict`、`Path=/`，过期时间不超过 72 小时和服务端 Session 到期时间。

Cookie 令牌使用高强度随机值，数据库只保存令牌哈希。邀请码也只保存哈希，明文只在后台创建成功时展示一次。

### 5.3 私有文件存储

加密 PDF 存放在发布目录、`public/` 和 Git 工作区之外的持久化私有目录，例如：

`/opt/revolution/shared/private/resume/`

应用通过只对 Web 和受控管理任务开放的私有挂载访问该目录。Edge、Embedding、Worker 的非必要角色不得获得文件读取权限。

文件采用 AES-256-GCM 加密。加密信封包含格式版本、密钥版本、随机 nonce、认证标签和密文；密钥通过生产 Secret 注入。数据库只保存文件元数据、密文校验值和当前文档指针，不保存 PDF 明文或可直接访问的公开路径。

### 5.4 文件返回

`GET /api/resume/file` 每次执行以下步骤：

1. 验证简历 Session、邀请码停用状态和 Session 到期时间；
2. 查询当前简历元数据；
3. 读取加密文件并完成 AES-GCM 完整性验证；
4. 成功后返回 `application/pdf`；
5. 记录一次成功的 PDF 返回审计事件。

响应至少设置：

- `Cache-Control: private, no-store, max-age=0`；
- `Pragma: no-cache`；
- `X-Content-Type-Options: nosniff`；
- `Content-Disposition: inline`；
- 不生成签名 URL，不经过公共 CDN 缓存。

授权用户可以通过浏览器 PDF 查看器打印或保存该最终 PDF。系统不尝试伪造浏览器端打印/保存审计，也不能阻止授权用户截图或主动转发。

## 6. 数据模型

### 6.1 `resume_documents`

保存唯一当前简历的非正文元数据：

- 文档 ID；
- 加密文件内部路径；
- 密文 SHA-256；
- 明文文件大小和密文文件大小；
- 加密格式版本与密钥版本；
- 上传时间、上传管理员和启用时间；
- 当前文档标记。

表中不得保存 PDF 明文、提取文本、联系方式或公开 URL。

### 6.2 `resume_invites`

- 邀请码 ID 与不可逆哈希；
- 熟人备注；
- 创建时间、7 天到期时间；
- 是否有效；
- 首次兑换时间；
- 停用时间与停用管理员。

每个邀请码最多成功创建一个简历 Session。并发兑换必须在数据库事务和行锁内完成，避免同一码产生多个 Session。

### 6.3 `resume_sessions`

- Session ID；
- 邀请码 ID；
- 令牌哈希；
- 创建时间、最后访问时间和 72 小时到期时间；
- 停用时间；
- 首次兑换 IP、User-Agent 和设备信息。

停用邀请码时必须在同一受控操作中停用关联 Session。文件接口每次读取 Session 和邀请码的当前状态，不能只依赖 Cookie 到期时间。

### 6.4 `resume_access_events`

记录：

- 邀请码创建、兑换成功、兑换失败、文件返回成功、退出、停用和过期清理；
- 原始 IP、User-Agent、设备信息、时间和结果代码；
- 关联邀请码与 Session 的内部 ID。

日志不保存邀请码明文、Session 令牌、加密密钥、PDF 正文或个人简历字段。超过 30 天的事件由 Worker 自动删除。

## 7. 管理后台

现有私有管理后台新增“简历管理”：

- 上传或替换当前 PDF；
- 查看当前文件的大小、密文校验值、上传时间和状态；
- 创建一人一码并一次性复制明文；
- 查看邀请码备注、兑换状态、到期时间和访问事件；
- 停用邀请码并立即撤销关联 Session；
- 查看 30 天内的 IP、User-Agent 和设备审计信息。

所有写操作必须经过现有管理员鉴权、密码复验或等价高风险确认、CSRF 防护、请求大小限制和统一审计。后台页面不得显示加密密钥或把 PDF 正文写入页面日志。

## 8. 上传与原子替换

后台上传只接受管理员提供的最终 PDF：

1. 限制请求大小，默认上限 10 MiB；
2. 同时校验扩展名、声明 MIME 和 `%PDF-` 文件头；
3. 不提取正文，不执行 OCR，不生成缩略图，不调用模型；
4. 使用随机 nonce 加密到同一私有文件系统中的临时文件；
5. 校验密文可完整解密且 SHA-256 一致；
6. 原子更新当前文档指针并切换加密文件；
7. 新版启用成功后删除旧版加密文件。

如果数据库提交、文件重命名或旧文件清理任一步骤失败，系统必须保持至少一份可识别的当前加密文件，并记录受控恢复事件。不得在私有目录或临时目录留下明文 PDF。

上传新版不会撤销现有有效 Session；文件接口始终解析当前文档指针，因此这些 Session 刷新后自动看到新版。

## 9. 失败与恢复

- 无效、已用、已过期或已停用的邀请码统一返回相同的未授权响应。
- 连续兑换失败沿用现有来源限速、失败窗口、锁定和异常告警机制，但使用独立的简历访问 scope。
- 文件缺失、密文损坏、密钥错误或认证标签失败统一返回不可用状态，不输出内部路径和加密细节。
- 加密密钥丢失时，现有文件不可恢复，只能由管理员重新上传最终 PDF。
- 密钥轮换必须先使用新密钥重新加密当前 PDF，验证成功后再切换密钥版本；不能直接覆盖旧密钥。
- Worker 负责清理过期 Session、过期邀请码、30 天以上审计事件和确认无引用的旧密文。

## 10. RAG 与对话隔离

公开知识摄取继续只读取获批的 `content/site-content.json`，且公开 profile 不得包含私密联系方式和简历经历。

简历模块不提供文本解析接口，不创建知识文档，不调用 Embedding，不向 Chat Service 暴露文件读取方法。Chat Service、JD Match、SearchRouter、管理导出和 Provider 适配层均不得引用 `resume_documents` 或私有文件目录。

安全测试必须拦截模型请求并断言：无论请求携带聊天 Cookie、简历 Cookie 还是两者同时存在，Provider 输入中都不包含 PDF 字节、简历字段、私有路径或熟人备注。

## 11. 验收合同

### 11.1 访问控制

- 未授权的 HTML、RSC、JS、API 和静态资源不包含简历正文或 PDF 字节。
- 聊天邀请码不能访问简历；简历邀请码不能访问聊天。
- 简历邀请码 7 天后不可兑换，只能成功兑换一次。
- 每个邀请码最多创建一个简历 Session；正常流程中，该 Session 只保存在完成兑换的浏览器中并持续 72 小时。
- 后台停用邀请码后，关联 Session 下一次请求立即失败。
- 上传新版后，已有有效 Session 返回新版 PDF，旧密文最终被删除。

### 11.2 加密与文件安全

- AES-256-GCM 正常加解密通过。
- 篡改密文、错误密钥、错误 nonce、错误认证标签和截断文件均拒绝返回。
- 非 PDF、伪造 MIME、损坏文件和超限文件均拒绝上传。
- 私有目录不存在明文 PDF，构建产物和镜像不存在简历文件。
- PDF 响应包含 `no-store`、`nosniff`、私有鉴权和内联响应头。

### 11.3 隐私与模型隔离

- `extractPublicKnowledge`、生产摄取和向量库中不存在简历正文与私密字段。
- 所有 Chat Provider 请求都不包含简历内容，即使同时持有两种授权。
- 自动化夹具只使用虚构 PDF，不把真实简历写入测试、截图、日志或证据目录。
- 生产验收只记录权限结果、密文校验值、文件大小和 HTTP 状态，不读取或输出真实正文。

### 11.4 审计与界面

- 成功文件返回、兑换、退出、停用和失败尝试均按合同记录。
- 原始 IP、User-Agent 和设备信息只在管理员后台可见，30 天后自动删除。
- 不声称记录真实打印或保存次数。
- 1440 和 390 宽度完成入口、验证、PDF 查看、无简历、过期和撤销状态验收。
- 浏览器控制台、页面错误和意外外部请求为零。

## 12. 非目标与明确边界

- 不解析 PDF，不自动生成网页简历，不执行 OCR，不接入 RAG 或模型。
- 不支持多份岗位定制简历，只维护一份当前 PDF。
- 不保存或分发 DOCX 等源编辑文件。
- 不添加水印。
- 不承诺阻止已授权用户截图、打印、保存或转发。
- 不承诺抵御已授权设备被完全控制或 Session Cookie 被设备级恶意软件窃取；`HttpOnly`、`Secure`、短有效期和立即撤销用于缩小风险窗口。
- 不把原始 IP 和设备日志长期保留，固定 30 天删除。
- 本设计不授权立即修改生产、上传真实简历、创建真实邀请码、推送或部署；实施和上线需经过后续计划、测试与单独授权。
