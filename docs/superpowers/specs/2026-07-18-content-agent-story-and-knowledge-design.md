# 内容创作 Agent 项目简介与知识库设计

日期：2026-07-18
状态：已实现并通过本地验收（尚未部署）

## 1. 目标与范围

本轮只完善“内容创作 Agent 系统”的公开项目简介、数字分身知识结构和后续简历复用事实，不顺带改写其他项目详情。

目标是建立两层体验：

1. 作品集用 20 秒可读完的简介吸引招聘方或潜在客户。
2. 数字分身用可检索的结构化知识回答模型能力、使用方式、工程亮点、个人贡献和未来方向。

本轮不实现自进化 Agent，不新增模型接入，不公开内部系统入口，也不制作最终简历。

## 2. 全局事实口径

- 摩斯展示的所有项目均由摩斯作为唯一开发者完成全部技术实现。
- 对外统一表述为：“本站展示项目的业务需求、产品方向和部分创意来自与业务方、使用者及相关人员的沟通；摩斯作为唯一开发者，独立完成全部技术实现。AI 工具用于研究、编码和验证，不构成人类协作团队。”
- 内容创作 Agent 的具体表述为：“项目的业务目标、需求和产品想法来自真实业务对接；摩斯负责把这些输入转化为可运行系统，并独立完成 Agent 编排、多模型接入、前后端实现、任务系统、部署交付与故障恢复。”
- 不得表述为摩斯独自提出全部业务需求、产品方向或创意，也不得抹去业务方和使用者的输入。
- 现有“逐系统协作边界待补齐”等旧口径属于过期事实，实施时必须从公开内容和知识源中移除。

## 3. 公开项目简介

页面首层只保留项目名称、一段简介、四个亮点、一句未来方向和一张获批设计图，不把完整技术栈或长篇工程说明堆在首层。

### 项目名称

内容创作 Agent 系统

### 一句话简介

一套面向电商内容生产的多模态创作系统。用户可以像使用 GPT 一样，通过对话调用 GPT Image 2、Seedance 2 等模型生成图片和视频。

### 四个核心亮点

1. GPT 式对话创作。
2. GPT Image 2 / Seedance 2。
3. 多参考图生成。
4. 任务恢复与资产管理。

### 未来方向

向自进化 Agent 系统演进：基于历史任务、人工反馈和质量评估，持续优化提示词、模型选择与创作策略，让系统越用越懂业务。

该段必须标注为未来方向，不得写成当前已实现能力。

### 展示媒体

- 只展示已获批的黑金新版主页面设计图。
- 图注继续说明“示例数据 / 不是生产运行截图”。
- 不增加旧版生产界面、真实客户素材或 Provider 配置截图。

## 4. 模型能力披露

模型名称是产品吸引力的证据，不是项目的主叙事。公开内容按成熟度分层：

- 真实链路已验证：GPT Image 2、GPT Image 2 Edit、Seedance 2 Fast。
- 已接入并可配置：Nano Banana 2、Gemini 2.5 Flash Image Edit、Veo 3.1、Kling、PixVerse V6、Wan。
- 条件边界：实际可用性依赖 Provider 账号、额度和模型权限；不得暗示所有模型在任何环境都稳定可用。

不得公开 API Key、内部 API 地址、workflow ID、任务 ID、费用、客户信息或真实业务数据。

## 5. 数字分身知识结构

内容创作 Agent 不再只生成一份长知识文档。公开事实源为项目提供六个独立知识主题，每个主题使用独立 document id，并共同链接到 `/works#content-agent`。

1. `project-content-agent-overview`：项目定位、目标用户、内容类型和业务价值。
2. `project-content-agent-experience`：GPT 式对话、参考图、确认生成和结果复用的用户流程。
3. `project-content-agent-models`：模型矩阵、成熟度和 Provider 条件边界。
4. `project-content-agent-engineering`：多参考图、能力约束、异步任务、失败恢复和资产归档。
5. `project-content-agent-role`：业务与产品协作输入、摩斯作为唯一开发者的技术实现范围、关键判断和工程贡献。
6. `project-content-agent-roadmap`：自进化 Agent 的计划、反馈闭环和明确未实现边界。

采用独立主题而不是继续扩大单一文档，是因为当前检索会对每个 document id 只返回最相关的一个 chunk。主题拆分后，数字分身可以在同一回答中获取多个项目侧面，同时保持来源可追溯。

## 6. 数字分身对话入口

项目简介下方增加一个站内对话入口，不作为内部系统访问按钮：

> 想了解它如何选择模型、处理参考图或恢复失败任务？问数字摩斯。

入口应打开现有数字分身，并携带项目上下文或预填问题“我想了解内容创作 Agent 系统”。推荐追问包括：

- 这个系统为什么不只是 API 套壳？
- 用户怎么通过对话生成图片和视频？
- 哪些模型已经真实跑通过？
- 摩斯在项目中具体负责什么？
- 自进化 Agent 下一步准备怎么做？

知识内容保持中立事实，回答层根据现有 audience intent 调整重点：招聘方优先个人贡献和工程证据，合作方优先使用场景与交付价值，同行优先架构取舍和边界。

## 7. 单一事实源与下游复用

项目事实继续以 `content/site-content.json` 为唯一公开事实源。实施时为项目补充结构化知识主题，不另建一套会漂移的营销文案数据库。

同一事实源产生三个下游表达：

- 作品集：简短简介、四个亮点、未来方向和对话入口。
- 知识库：六个可独立检索的主题文档。
- 简历：项目定位、业务协作背景、个人独立技术实现范围、核心模型能力和两至三条工程成果。

未来完善其他系统时复用同一结构，但仍坚持一个项目一个项目采集、核验和上线。
