# Chat v2 对话路由、回答质量与响应可靠性修正规格

日期：2026-07-22
状态：设计已确认，等待书面规格复核
执行合同：`STAGED / CRITICAL / DEPLOYED`
授权边界：本规格不授权真实 Provider 调用、扩大灰度、push 或部署

## 1. 适用范围与覆盖关系

本规格修正 `2026-07-21-digital-morse-conversation-v2-design.md` 中已经被真实评审证明不合适的对话路由、人格上下文、回答质量、响应可靠性与客户端等待策略。数字 Morse 的固定身份、证据型候选人陈述、私密简历隔离、灰度和评测原则继续有效。

以下旧条款由本规格覆盖：

- 第 8 节“守卫失败后安全证据模式”的自动本地回答。
- 第 5-6 节中把完整身份卡注入所有人格、以及无法识别时默认进入项目 RAG 的实现解释。
- 第 9.2 节 `0 / 8 / 14` 秒错峰并发接管。Hedging 继续关闭，不在本轮启用。
- 第 9.3 节客户端自动重连和 Provider 全失败后的本地摘要回答。
- 第 11 节过弱的等待反馈。
- 第 13 节中不能区分协议事件、模型正文和用户可见正文的首字节指标。

本规格不改变生产 canary 范围，也不继续已经暂停的真实 Provider 20 轮评审。实施和确定性验证完成后，真实评审必须重新获得授权。

## 2. 已确认的问题与根因

生产记录和主中转面板共同证明，主中转本身并非所有慢请求的主要耗时来源：正常请求约 2-9 秒完成，应用层随后仍可能等待数十秒甚至数分钟。

直接根因如下：

1. 当前 `social` 路由只识别少量完整匹配的“你好、谢谢、再见”。任何没有命中关键词的问题都会默认成为 `project + grounded + RAG`，所以“今天吃饭了吗”也会检索作品集。
2. 当前只要出现“岗位、匹配、招聘”等词就进入招聘 RAG，没有区分“已经提供 JD”和“只是要求岗位适配分析”。没有 JD 时仍会拿项目片段生成看似正式的模板回答。
3. 当前人格提示无论意图都附带完整 `approved_identity_card` 和所有项目摘要。即使路由到 social，模型仍持续看到作品集清单，容易主动把闲聊拉回项目和能力介绍。
4. 当前输出守卫只检查引用、招聘禁语、下一步和开发助手口吻，没有“是否直接回答当前问题”或“是否沿用错误模板”的质量条件。
5. Responses 流收到任意协议事件后就取消首字节计时。`response.created`、推理阶段或其他无正文事件被误认为用户已经得到正文，空流因此可继续等满 90 秒。
6. Provider 超时、网络故障和输出守卫拒绝都可以触发 strict 重生成。一次线路故障可能重新执行整套 Provider 协调。
7. strict 重生成拥有新的 Provider 总超时，没有继承原轮次剩余预算。
8. 客户端对可恢复错误最多自动 replay 三次，服务端的多次 attempt、strict 和客户端 replay 发生乘法叠加。
9. Provider 全失败后，本地安全回答拼接 top 2 检索片段，没有验证它们是否回答当前问题，因此不同问题可能得到相同且答非所问的回复。
10. 招聘评审样本缺少 JD，却要求判断岗位匹配度，输入本身无法支持有效回答。
11. 前端已有阶段文案和静态思考点，但视觉反馈过弱，用户会把等待理解为页面卡死。

## 3. 目标与非目标

### 3.1 目标

- 让数字 Morse 首先成为可以自然连续交流的数字分身，而不是作品集 RAG 查询器。
- 默认进入自由对话；只有当前问题明确需要 Morse 的公开事实、项目证据或已提供 JD 时才启用 RAG。
- 闲聊、一般观点和通用技术讨论不加载项目摘要、不强制引用、不把话题拉回作品集。
- 多轮追问能继承上一轮明确主题；用户切换话题时立即重新路由，不被首次入口或旧意图锁定。
- 主线路获得足够的正常推理时间，不因固定 20 秒门槛被过早中断。
- 所有线路、strict 和恢复动作共享一个有界总预算，禁止串行超时无限累加。
- 网络或 Provider 故障只触发线路切换；strict 只修复已生成但未通过输出守卫的回答。
- Provider 全失败时明确失败、不扣消息次数、不生成无关本地回答。
- 缺少 JD 时先补齐输入，不消耗 Provider 调用。
- 等待期间持续提供明显、轻量且可停止的活动反馈。
- 记录足以区分中转慢、模型推理慢、应用缓冲慢和守卫重生成慢的遥测。

### 3.2 非目标

- 不启用 hedging，不同时向两个 Provider 发送同一问题。
- 不增加运行时 LLM judge；回答相关性不能通过额外一次模型调用换取。
- 不让模型在自由对话中编造真人 Morse 的实时位置、饮食、情绪、经历或其他未提供的现实状态；数字分身可以自然回应，但必须保持身份真实。
- 不伪造确定进度百分比，不展示模型思维链。
- 不重做聊天整体布局，不新增前端依赖。
- 不修改私密简历、管理员认证、邀请码或 Provider 密钥管理边界。

## 4. 方案选择

### 4.1 固定加长每条线路

把每个节点从 20 秒改为 40-60 秒可以减少过早切换，但主、备、strict 和客户端 replay 仍会串行累加。拒绝。

### 4.2 恢复错峰并发

Hedging 可以降低尾延迟，但会增加成本、并发和重复候选处理复杂度，且本轮用户明确要求不扩大灰度。拒绝。

### 4.3 共享总预算下的自适应串行接管

采用。主线路使用软门槛和硬门槛区分“完全无响应”与“协议仍活跃但尚未产生正文”；所有 attempt 和 strict 共享一个轮次 deadline。该方案同时限制过早切换和最坏等待时间。

## 5. 时间预算与定义

### 5.1 两层 deadline

- 交互轮次 deadline：从服务端接受合法请求开始最多 90 秒，覆盖路由、Embedding、检索、可选搜索、Provider、守卫和提交。
- Provider 阶段 deadline：进入回答生成后最多 80 秒，并受交互轮次剩余时间限制。

Provider、strict 或 failover 不得创建新的独立 80 秒预算。它们只能消费同一绝对 deadline 的剩余时间。

### 5.2 单次 attempt 的软门槛与硬门槛

- `25 秒软门槛`：当前线路完全没有有效协议事件时中止并串行切换下一条健康线路。
- `40 秒正文硬门槛`：25 秒前已经收到有效协议事件，但服务端仍未收到第一段模型正文时，最多继续等待到 attempt 启动后的第 40 秒。
- 协议活动只能触发一次从 25 秒到 40 秒的延长。后续元数据、心跳或重复状态事件不能重置计时器。
- 主线路、备用 normal 和备用 strict 使用同一规则；实际门槛取本节门槛与 Provider 阶段剩余时间的较小值。
- 剩余 Provider 阶段预算少于 10 秒时不再启动新 attempt。
- 网络拒绝、明确 4xx/5xx、流错误和连接失败立即结束当前 attempt，不等待软门槛。
- 服务端收到模型正文后取消首正文门槛，但仍受 Provider 阶段和交互轮次 deadline 约束。

“协议事件”“模型正文”和“用户可见正文”必须分开定义：

- 协议事件：Provider 流已建立并产生合法 Responses/Chat Completions 事件，不代表用户已经看到内容。
- 模型正文：服务端收到非空文本 delta 或完整文本事件。
- 用户可见正文：通过当前缓冲和输出守卫后第一次写入 SSE `delta`。

招聘和 JD 使用完整缓冲时，模型正文可以取消 40 秒门槛，但等待 UI 必须持续到回答通过守卫并真正可见。

### 5.3 已经开始可见输出后的行为

普通回答一旦向用户释放正文，就锁定当前 Provider，不再把备用候选拼接到已显示文本后。若后续流中断，保留稳定失败状态和手动重试入口，不自动重放整轮。

尚未向用户释放正文的完整缓冲回答可以在当前 Provider 失败后切换备用，因为不会产生候选文本混合。

## 6. 单一服务端状态机

```text
请求校验
  -> 结合当前问题与上一轮明确主题做本轮路由
      -> 自由对话：不检索，直接以数字分身人格生成
      -> 外部实时信息：只执行受控联网，不加载个人知识
      -> 身份介绍：只加载精简身份资料
      -> 个人能力核验：先查询能力台账，必要时补充已准入项目证据
      -> 公开项目：条件检索与当前问题相关的公开证据
      -> 通用技术讨论：不检索；询问 Morse 的具体实现时才检索
      -> 缺少 JD 的岗位匹配请求：要求提供 JD，不调用 Provider，不扣次数
      -> 已提供 JD：结合能力台账和相关公开证据执行匹配
  -> 主线路 normal
      -> 基础设施失败且尚无可见正文：备用线路 normal
      -> 输出守卫通过：提交并完成
      -> OUTPUT_GUARD_REJECTED：同线路 strict
          -> strict 通过：提交并完成
          -> strict 基础设施失败或再次被拒绝：备用线路 strict
  -> 共享 deadline 或 attempt 上限耗尽：明确失败，不扣次数
```

### 6.1 attempt 上限

一轮最多 3 个真实 Provider attempt。normal、strict 和 failover 都计入同一个上限。

典型路径：

- `主 normal -> 完成`：1 次。
- `主 normal 基础设施失败 -> 备 normal -> 完成`：2 次。
- `主 normal 基础设施失败 -> 备 normal 被守卫拒绝 -> 备 strict`：3 次。
- `主 normal 被守卫拒绝 -> 主 strict 失败 -> 备 strict`：3 次。
- `主 normal 失败 -> 备 1 normal 失败 -> 备 2 normal`：3 次，不再 strict。

没有足够剩余时间启动一个有意义的 attempt 时直接终止，不为凑满次数启动注定被立即取消的请求。

### 6.2 错误分类

仅 `OUTPUT_GUARD_REJECTED` 可以触发 strict。以下错误不得触发 strict：

- 首协议事件超时、首模型正文超时或共享总超时。
- DNS、连接、TLS、网络中断、429、5xx 和 Provider stream failed/incomplete。
- 用户停止、Session、额度、输入校验、Embedding、检索或搜索错误。

守卫拒绝证明 Provider 可达，不计入节点基础设施熔断失败；网络和协议失败继续进入节点健康统计。

## 7. 数字分身对话、回答正确性与降级

### 7.1 产品行为原则

数字 Morse 的主循环是“理解当前这句话并自然回应”，不是“每轮检索作品集并套证据模板”。RAG 是事实工具，不是默认人格。

每轮必须独立回答三个问题：

1. 用户现在是在和我聊天，还是在询问我的公开事实？
2. 当前问题是否真的需要站内证据才能回答？
3. 如果需要证据，缺少的输入是否已经具备，例如岗位适配所需的 JD？

无法高置信判断时默认自由对话或提出一个自然的澄清问题，不得默认进入项目 RAG。

### 7.2 本轮路由合同

| 当前问题 | 路由 | 证据 | 行为 |
| --- | --- | --- | --- |
| 问候、吃饭、近况、兴趣、感受、轻量观点 | `conversation` | `none` | 正常聊天，不引用、不介绍项目 |
| 通用知识或通用技术讨论 | `conversation` | `none` | 直接讨论问题，不自动声称是自己的项目经验 |
| 明确要求最新、当前或外部事实核验 | `external_current` | `web` | 只执行受控联网，不加载个人项目；无法联网时明确不能完成时效核验 |
| “你是谁、你擅长什么” | `identity` | `identity` | 只加载精简身份资料，按问题控制篇幅 |
| 明确核验某项个人能力或经验 | `personal_fact` | `ledger` | 先查询结构化能力台账；需要项目叙述时才补充检索 |
| 明确询问 Morse 的项目、做法或结果 | `grounded` | `rag` | 只检索与当前问题相关的公开证据 |
| “给我岗位适配度、分析我是否适合这个岗位”但没有 JD | `jd_intake` | `none` | 要求提供 JD，不调用生成或检索 Provider |
| 已提交完整 JD 或显式使用 JD 工作流 | `jd` | `ledger + rag` | 按 JD 要求匹配已批准能力与公开证据，完整缓冲后回答 |

现有 `social` 路由替换为 `conversation`。该路由必须覆盖开放式日常交流，不能继续依赖只匹配“你好/谢谢/再见”的窄正则。

关键词只能作为高置信路由信号，不能作为最终答案。例如出现“岗位”不等于已经具备 JD；出现“技术”也不等于必须检索个人项目。

### 7.3 路由、条件影子检索与调用次数

采用“策略路由 + 结构化能力台账 + 条件检索 + 一次最终 LLM”，不采用默认 RAG，也不采用所有问题都执行 Embedding 的全量影子检索。

1. 显式 `jd_match`、结构化 JD、无 JD 匹配请求和身份介绍由确定性前置门直接处理。
2. 策略路由器结合当前问题中的对象、谓词和问题形态判断 `conversation / external_current / identity / personal_fact / grounded / jd_intake / jd / clarify`。单个关键词不得单独触发个人事实模式。
3. 日常交流、稳定通用知识、通用技术和职场建议进入 `conversation`，不调用 Embedding、RAG 或 Search。
4. 只有明确要求最新信息、外部资料或事实核验时才进入 `external_current`，直接复用受控 SearchRouter 与搜索额度，不先执行个人知识 Embedding。网页结果只能回答外部问题，不能补充 Morse 的经历、能力、项目状态或其他个人事实。
5. `personal_fact` 先查询本地结构化能力台账。仅当回答需要项目过程、取舍或结果叙述时，才并行准备相关公开片段；未被证据准入器采用的候选不得进入最终提示。
6. `grounded` 和 `jd` 才执行本地 Embedding 与 pgvector 候选检索。检索是影子阶段：候选在准入完成前不可见，也不能自动注入最终 LLM。
7. 无法高置信区分“通用讨论”与“个人事实核验”且差异会影响事实声明时，进入 `clarify`，只提出一个自然澄清问题，不生成个人经历。
8. 正常回答只有一次生成式 LLM 调用。不得新增“LLM 路由一次 + LLM 回答一次”或运行时 LLM judge；`jd_intake` 和确定性澄清可以零次生成式调用。

策略路由器负责判断是否需要个人事实；证据准入器负责判断候选属于 `direct / transferable / none`；最终 LLM 只负责自然表达，不能改变路由或提升证据等级。

### 7.4 多轮上下文与话题切换

- 当前问题明确、自包含时，只按当前问题路由，旧的 `audienceIntent`、入口按钮和上一轮意图都不能覆盖它。
- 当前问题是“这个怎么做的、为什么这样选、那结果呢”等短追问时，可以继承上一轮明确的项目、技术或 JD 主题。
- 继承只允许一轮，并且必须由代词、承接词或短追问结构触发；新主题出现时立即清除继承。
- 需要 RAG 的承接追问使用“上一轮明确主题 + 当前问题”构造检索查询，避免只检索“这个怎么做”这类无主题文本。
- 对话历史用于保持语义连续，不得把旧回答当成当前问题的模板重复输出。

### 7.5 路由锚点与持久化

不能只从上一条自然语言重新推导继承主题。`interaction_turns` 使用 additive nullable 字段或等价独立记录保存：

- `route_kind`：本轮最终路由。
- `route_reason_code`：稳定、非敏感的决策原因码。
- `topic_kind`：`none / external / project / capability / jd`。
- `topic_ref`：公开项目 slug、规范化能力 ID 或受控 JD 标识；不得保存新的原始 JD、私密简历片段或自由文本摘要。
- `evidence_class`：`none / identity / web / direct / transferable / mixed / unavailable`。
- `inherited_from_turn_id`：仅在合法的一轮承接中记录来源 turn。

当前问题自包含时必须忽略上一轮锚点。只有代词、承接词或短追问结构命中时，才读取上一轮已持久化锚点；不得通过重新执行最新版路由规则来猜测旧 turn 的含义。重试同一 `turnId` 必须复用已记录决策，避免一次请求在不同 attempt 之间改变事实范围。

### 7.6 人格与证据上下文分离

所有轮次只共享精简固定身份：数字 Morse、第一人称、自然交流、不得编造和泄密。

- 自由对话只注入精简固定身份，不注入完整身份卡、项目列表、RAG 来源或招聘规则。
- 身份问题只注入公开定位和最多两个与问题相关的代表项目，不附带全部项目摘要。
- grounded/recruitment/JD 只注入本轮检索命中的证据，不把未命中的项目作为背景噪声。
- 通用技术讨论可以使用模型常识，但不能把常识表述为 Morse 已完成的项目事实。

人格负责“怎么说”，证据负责“哪些个人事实可以说”。两者必须是独立输入，不能再用完整作品集卡片定义每一轮人格。

### 7.7 结构化能力台账与证据准入

`content/site-content.json` 中现有项目职责、`capabilities`、`techStack`，以及 `profile.resumeFacts` 中人工脱敏并批准公开的职业事实，是个人事实的公开来源。原始私密简历、联系方式、真实公司/学校名称和存储元数据不得进入公开知识。运行时不得临时解析叙述来推断能力等级，而是生成候选后由一份受版本控制、人工审核并经过交叉校验的能力台账批准：

- 能力 ID 与显示名称，例如 `kubernetes`、`docker-compose`。
- 仅做词形统一的别名，例如 `k8s -> kubernetes`、`postgres -> postgresql`；别名不得表达技术迁移关系。
- `direct` 项目引用、事实范围和验证等级，例如“实现过”“本地验证”“生产已观察”。技术栈出现不能自动升级为“精通”或“生产运维过”。
- `transferable` 必须是人工批准的显式映射，包含来源能力、目标能力和允许措辞。向量相似度、共同项目或技术邻近不得生成或提升该关系。
- 对应的项目 slug 或脱敏简历事实 ID、原始技术项和证据边界。构建期校验台账引用仍存在且没有被项目边界否定；失效引用必须使检查失败。

证据准入规则：

1. 规范化能力或批准别名命中台账且事实范围满足问题，才能判定 `direct`。
2. 没有 direct，但存在已批准迁移关系时判定 `transferable`；回答必须同时说明不能据此确认目标能力的直接实践。
3. 两者都没有时判定 `none`，不得用向量 top result、相邻技术、常识或网页搜索补造个人经验。
4. pgvector 的 `LOCAL_EVIDENCE_MIN_SCORE` 只代表语义候选相关，不代表直接经验。项目叙述还必须通过项目/topic metadata、问题焦点和公开边界校验后才能注入。
5. 关于 Morse 的个人事实继续禁止联网补证据；公开项目与脱敏简历事实没有提供的经历不能由外部网页补齐。“未检索到”只表示当前公开材料未提供，不得改写成“从未做过”。

Kubernetes 基准案例固定为：`direct = none`；Docker、Docker Compose、容器化部署和健康检查只能在存在显式迁移映射时作为 `transferable`。允许表达“公开项目能确认容器化部署基础，但不能据此确认 Kubernetes 生产实践，建议面谈核实”，禁止表达“具备 Kubernetes 实战”或把 Docker 自动改写为 Kubernetes。

### 7.8 删除自动本地拼接回答

Provider 全失败后不得把检索 top 2、身份卡或项目摘要自动拼成当前问题的回答。自动 safe fallback 从正常 Provider 失败路径移除，避免不同问题得到同一回复。

显式运维安全模式 `MORSE_CHAT_SAFE_MODE` 仍是独立人工开关，不属于 Provider 故障后的自动降级。

### 7.9 失败语义与额度

- 全部 Provider 失败时返回稳定错误状态：“这次线路没有完成回答，问题已经保留，可以重试。”
- 未提交完整回答不得扣消息次数，不写完成 assistant 历史，不把部分文本当成完成结果。
- 手动重试复用原 `turnId` 的幂等语义；服务端已经提交时重放已提交结果，未提交时创建新的受控执行。
- 客户端删除 Provider 错误的自动 replay 循环，只保留用户主动重试。

### 7.10 缺少 JD 的确定性前置门

自由对话中出现明确的岗位匹配、胜任度或 JD 分析请求，但没有本轮或受控工作流 JD 时：

- 不执行 Embedding、RAG、搜索或 Chat Provider。
- 返回简短引导，要求粘贴职位描述，并引导进入现有 JD 匹配入口。
- 不生成匹配结论，不扣消息次数。

固定 20 轮评审中的岗位匹配样本必须携带 `workflow=jd_match` 和实际测试 JD；缺少 JD 的样本只能测试前置门，不能计入回答质量分数。

### 7.11 完整 JD 的正向陈述规则

完整 JD 回答优先呈现能够由公开证据确认的匹配内容，并把直接证据与可迁移证据区分开：

- 先回答最相关的项目、职责和已验证能力，不输出匹配百分比，不使用“明显不匹配”或完整缺口清单。
- 没有 direct 且未被单独追问的能力默认不主动展开，不得生成审计式边界段落或负面能力列表。
- 用户单独追问某项能力时必须直接、简短回答该项事实边界，不能为了正向表达而回避问题；允许补充已批准的 transferable，但必须明确它不是 direct。
- “未主动展开”不等于已经具备；“当前公开材料未提供”也不等于从未做过。

### 7.12 问题相关性与模板感门禁

- 提示词将当前问题和本轮回答目标放在历史记录之后，明确要求第一段先直接回应当前问题；只有 grounded/JD 才在后续补证据。
- 只把通过相关度过滤的公开证据交给生成器；不得用相邻项目填满回答。只有对方明确核实个人事实时才简短说明该项暂未能确认，普通项目/JD 回答不主动输出边界段落。
- 自由对话出现来源引用、项目清单、岗位分析结构或“根据资料”式措辞时直接判为路由/人格失败。
- personal_fact/grounded/JD 回答如果没有回答当前问题，只是在罗列项目摘要，则由输出守卫以 `answer_not_direct` 拒绝并触发唯一一次 strict 重生成。
- personal_fact 回答必须提到本轮规范化能力或批准别名；被明确追问的 `none` 与 `transferable` 必须包含该项事实边界，不能用含糊措辞暗示 direct。JD 中未被单独追问的 `none` 不要求输出。
- 同一会话中，不同问题得到完全相同或高度相似的长回答时，以 `template_repetition` 拒绝；该检查是防回归手段，不是本次设计的主要路由机制。
- 不在热路径新增第二次 Provider 相关性评审。
- 真实 20 轮评审必须分别记录“像不像自然对话”“是否直接回答问题”“是否错误调用 RAG”“证据是否支持”。模板化、答非所问或无理由 RAG 任一出现都判本轮失败。

## 8. SSE、遥测与可观察性

### 8.1 SSE 状态

继续使用现有 `routing / knowledge / web / answering / switching / handoff` 状态，不暴露 Provider 域名、节点名、错误码、attempt 数或内部提示词。

- 初始等待使用 `answering` 或中性“正在处理你的问题”。
- 串行切换时发送一次 `switching`。
- SSE heartbeat 只维持连接，不影响任何 Provider deadline。

### 8.2 attempt 遥测

每个 attempt 至少记录：

- `started_at`、`first_protocol_event_ms`、`first_model_text_ms`、`first_user_visible_ms`。
- 真实 wall duration、完成状态、稳定错误码、normal/strict 和 primary/failover 启动类型。
- usage、成本完整性和赢家信息；未知值保持 `null`，不得推断。

轮次同时记录从请求接受到终态的真实 wall duration。数据库时长、Provider attempt 跨度和公网观察必须能互相对齐。原始问题、JD、回答、Provider URL、Key 和系统提示不得进入遥测。

## 9. 等待交互

等待体验是可靠性反馈，不承担掩盖故障的职责。

- 待回复的 assistant 消息内显示一个非确定进度条，持续移动但不显示虚假百分比。
- 进度条下复用当前 SSE 阶段文案；没有阶段事件时显示“正在处理你的问题”。
- 8 秒后显示已等待秒数；30 秒后使用低干扰提示“仍在处理中，你可以继续等待或停止”。
- 进入 `switching` 时明确显示线路切换状态，避免用户把状态变化误认为重新开始。
- 保留当前停止按钮；停止后进度条立即消失并显示可手动重新生成状态。
- 进度条拥有固定最小高度，阶段切换不引起消息区跳动。
- `prefers-reduced-motion` 下停止位移动画，使用静态高对比活动标记。
- 390 宽度允许文案换行，不遮挡输入框、停止按钮或已生成内容。

不实现历史状态时间线、滚动内部步骤、假的“正在执行命令”或模型思维链。

## 10. 实施边界

实现应保持以下边界独立：

1. 对话策略器：只根据当前问题、显式工作流和合法的一轮主题锚点产生路由与稳定原因码。
2. 能力台账与证据准入器：规范化别名，区分 direct/transferable/none，并过滤检索候选。
3. Provider 流计时：区分协议事件、模型正文和总 deadline。
4. 回答执行协调：管理共享 deadline、attempt 上限、normal/strict/failover 状态机。
5. Chat 服务：处理缺少 JD、路由持久化、失败补偿、额度和 SSE 状态。
6. 客户端 hook：删除自动 replay，维护单个 pending assistant 和等待秒数。
7. 展示组件：非确定进度条、阶段文案、停止和 reduced-motion。
8. 遥测：additive nullable 字段或等价持久化，不改写历史 attempt。

不得把所有时序逻辑继续堆入 `chat-service.ts`；绝对 deadline 和 attempt policy 应由可独立测试的执行协调边界拥有。

## 11. 验收标准

### 11.1 确定性测试

- “今天吃饭了吗”“最近忙什么”“你怎么看 AI”进入 `conversation + none`，不调用 Embedding、RAG 或 Search，不向提示词注入项目列表。
- “职场里怎么和同事处理分歧”进入 `conversation + none`；“你以前怎么处理同事冲突”进入 `personal_fact`，无公开经历时说明边界，不编造案例。
- “Next.js 当前最新版本是什么”进入 `external_current + web`，不执行个人知识 RAG；搜索不可用时不得用模型记忆冒充已核验结果。
- “Morse 当前有哪些项目”仍进入个人事实路径并禁止联网补证据，不能因为包含“当前”而进入 `external_current`。
- “什么是 RAG”进入通用对话且不加载个人项目；“你的数字 Morse 怎么实现 RAG”进入 grounded 并检索相关公开项目。
- “Kubernetes 是什么”进入通用对话；“你有 Kubernetes 生产经验吗”进入 `personal_fact`，结果为 `direct=none`，只允许把已批准的 Docker/部署证据标为 transferable。
- “你用过 Docker Compose 吗”命中能力台账 direct，并返回对应项目证据；不得因为两者相关而把该结论扩展成 Kubernetes direct。
- “给我一份岗位适配度”在没有 JD 时进入 `jd_intake`，不调用生成、Embedding、RAG 或 Search Provider。
- 完整 JD 和显式 JD 工作流进入 `jd + ledger + rag`；回答必须针对 JD 要求，不得只罗列项目摘要。优先展开 direct 与已批准 transferable，不主动列出无 direct 项或完整缺口清单。
- “你用过 Cursor、CC 和 Codex 吗”进入 `personal_fact`；Claude Code/CC 与 Codex 命中脱敏简历 direct，Cursor 保持 `none`。回答只处理这三个被问到的工具，不能扩展其他能力边界。
- “这个为什么这样设计”能够继承上一轮明确项目主题；从项目追问切换到“今天吃什么”后立即回到 conversation。
- 自包含的新问题不得继承旧 route/topic；合法短追问只继承一轮，并记录 `inherited_from_turn_id`。同一 turn 重试不得改变 route/evidence 决策。
- conversation 提示只含精简数字分身身份，不含完整身份卡、项目清单、引用规则或招聘模板。
- conversation 回答不得出现来源引用、项目清单或岗位分析结构；grounded/JD 回答必须直接回应当前问题。
- pgvector 高分但能力台账没有 direct 的候选不得被表述为个人直接经验；台账引用失效、别名冲突或边界矛盾时构建检查失败。
- conversation、external_current、identity、personal_fact、grounded、jd_intake、jd 和 clarify 均有正例、近邻反例及跨轮切换测试；测试同时断言是否调用 Embedding/RAG/Search/Chat Provider。
- 25 秒内完全无协议事件时主线路被中止并串行切备用。
- 25 秒前有协议事件时可延长到 40 秒；重复元数据不能继续延长。
- 30-40 秒之间到达的第一段模型正文不会被 20 秒旧门槛误杀。
- 40 秒仍无模型正文时切换，所有后续 attempt 只使用 80 秒共享 deadline 的剩余时间。
- Provider 超时和网络错误不触发 strict；只有 `OUTPUT_GUARD_REJECTED` 触发。
- normal、strict 和 failover 合计不超过 3 个 attempt。
- 客户端一次提交只发起一次 `/api/chat` 请求，不自动 replay Provider 错误。
- Provider 全失败时没有本地拼接回答、没有扣次数、没有完成 assistant 历史。
- 无 JD 的岗位匹配请求不调用 Embedding、Search 或 Chat Provider。
- 两个不同问题不会因 safe fallback 得到同一个拼接回答。
- 停止、deadline、SSE 断开和幂等重试均保持额度与 turn 终态一致。

### 11.2 UI 验收

- 1440x900 和 390x844 均能看见 pending assistant 内的活动进度条、当前状态、等待时间和停止入口。
- 进度条在收到首个可见正文后退场，不遮挡流式文本。
- 切换线路、停止、失败和完成状态不发生布局跳动或重复消息。
- reduced-motion 下无持续位移动画。
- console/page error 为 0。

### 11.3 性能与真实评审

- 本地故障注入证明单轮 Provider 阶段不超过 80 秒，交互轮次不超过 90 秒。
- 生产评审分别报告端到端、首协议事件、首模型正文、首用户可见正文和各 attempt wall duration，不能再用“首字”混称。
- 真实 20 轮评审只能在重新授权后开始。固定组成是：6 轮自由对话与自然承接、3 轮通用知识/职场建议、3 轮身份/项目及跨轮追问、3 轮通用技术与个人实现对照、3 轮个人能力证据（含 Kubernetes none 与 Docker direct/transferable 对照）、2 轮招聘/JD（含 1 轮无 JD 前置门）。
- 每轮记录预期路由、实际路由、是否调用 RAG、是否直接回答、自然度、数字分身一致性、证据支持度和最终线路。预期 `evidence=none` 的轮次出现 RAG，或任一回答明显答非所问、像固定项目模板，均直接判该轮失败。
- 20 轮至少 18 轮通过；私密信息泄露、虚构个人事实、无 JD 生成岗位适配结论和自由对话错误调用 RAG 均为零容忍，出现一次即停止评审。
- 本轮不扩大 canary，不启用 hedging，不用 Mock 冒充真实 Provider 质量。

## 12. 发布与回滚

本地实现先通过 focused tests、受影响边界测试、构建和双宽浏览器验收。之后才可以单独请求真实 Provider 评审授权。push、生产部署和灰度扩大仍是独立授权动作。

新增计时和状态机必须可通过配置或版本开关回到当前串行 Provider 路径，但不得恢复客户端三次自动 replay 或自动拼接无关本地回答。出现重复扣次数、超出共享 deadline、私密内容泄露或 attempt 超上限时停止评审并关闭 Chat v2 canary。
