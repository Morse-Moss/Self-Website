# `/admin` OpenAI-Compatible API 管理设计

日期：2026-07-21

状态：设计已确认，待实施计划

预期实施合同：`STAGED / CRITICAL / LOCAL`

## 1. 目标

为 Revolution 的私有管理后台增加全站级 OpenAI-compatible API 管理能力。管理员可以在 `/admin` 中保存多个中转、维护模型预设、执行显式连接测试、排列一条主线路和最多五条备用线路，并在不重启服务的情况下原子切换新请求使用的对话模型与中转。

本设计只管理 Chat Provider。Embedding、BGE、pgvector 维度、知识摄取和 RAG 数据继续由部署环境控制，不能从该页面修改。

本轮只完成设计确认；未修改运行代码或数据库，未读取真实 API Key，未执行 Provider 调用，也未部署。

## 2. 当前状态与问题

当前 Chat 配置完全来自进程环境变量：

- `loadServerConfig()` 读取主线路的 API Key、Base URL、模型、协议和 reasoning effort；
- `createProvider()` 在每个 Chat 请求开始时创建 OpenAI-compatible 客户端；
- 最多两个备用节点只能覆盖 API Key 和 Base URL，所有节点共享同一模型、协议和输出限制；
- `/api/chat` 把 Provider 名固定记录为 `openai`，无法准确说明真正成功的备用线路；
- `/admin` 只提供对话复盘、邀请码和导出，没有 Provider 配置入口；
- PostgreSQL 没有 Provider 配置、活动版本、测试结果或配置审计表。

因此，当前切换模型或中转需要修改部署环境并重启进程，不能保存多套配置、测试后切换、原子回退或安全删除失效线路。

## 3. 已确认的产品规则

- 只有管理员可以管理和切换 Provider；访客聊天界面不展示模型或中转选择。
- 首版只支持 OpenAI-compatible 接口，不实现 Anthropic、Gemini 等原生协议适配器。
- 每个模型预设明确选择 `responses` 或 `chat_completions`。
- 后台保存多个中转和多个模型预设，不覆盖唯一一份当前配置。
- 活动路由固定为一条主线路和零至五条备用线路。
- 新增或发生运行参数变化的线路必须在最近 30 分钟内完成一次管理员主动触发的真实测试，才可激活。
- 保存配置不会自动调用外部 Provider。真实测试按钮必须明确提示可能产生极小费用。
- 新活动版本无需重启；新请求使用新版本，已经开始的流式请求固定使用开始时的配置快照。
- 只有当前线路尚未输出正文时才允许切换备用线路。
- 失效线路必须提供删除功能，同时保留必要的历史解释能力。
- 当前环境变量主节点和已配置的 fallback 节点分别保留为只读启动/应急目标。

## 4. 明确非目标

- 不让访客按会话或按消息选择模型。
- 不管理 Embedding Provider、Embedding 模型或向量维度。
- 不支持任意原生 Provider SDK、图片模型、视频模型或语音模型。
- 不从后台修改 `.env.production`、Docker Secret 或主机环境文件。
- 不自动扫描、轮询或定时探活真实 Provider，避免未授权费用。
- 不自动部署、推送、写入生产数据库或执行真实付费 smoke。
- 不在首版提供加密主密钥轮换 UI；密钥轮换属于受控运维流程。

## 5. 总体架构

### 5.1 环境启动层

以下配置继续由部署 Secret 注入：

- 数据库连接与数据库 TLS 配置；
- 管理员密码哈希、管理域名和 Session 策略；
- `MORSE_PROVIDER_CONFIG_KEY_FILE`，生产 Web 通过文件读取 Base64 编码的 32 字节随机密钥；development/test 可改用 `MORSE_PROVIDER_CONFIG_KEY` 直接值；
- `MORSE_PROVIDER_CONFIG_KEY_VERSION`，首版使用明确的正整数版本；
- 当前 `OPENAI_*` Chat 配置，作为只读环境应急线路；
- Embedding、搜索、预算门、超时和并发等非 Provider 凭据配置。

加密主密钥不得进入数据库、浏览器、日志、错误响应、构建产物或 Git。生产启动必须校验密钥格式；活动数据库配置无法解密时 readiness 失败。

### 5.2 数据库运行层

数据库保存：

- 版本化的中转连接；
- 版本化的模型预设；
- 不可变路由版本及有序目标；
- 当前活动版本指针；
- 连接测试和配置操作的安全审计事件。

环境主节点以及已配置的 `OPENAI_FALLBACK_1_*`、`OPENAI_FALLBACK_2_*` 分别作为运行时虚拟目标暴露给管理页，只显示非敏感元数据。管理员可以把任一环境目标放入主备顺序，但不能查看 Key、编辑或删除。

### 5.3 请求级配置快照

每次 `/api/chat` 开始时执行一次运行配置解析：

1. 读取当前活动路由版本；
2. 解析一至六个有序目标；
3. 在服务端解密数据库目标的 API Key；
4. 验证目标版本未删除且参数完整；
5. 构造请求专属 Provider Router；
6. 将路由版本和目标元数据固定到本次请求。

活动配置变更只影响之后开始的请求。正在输出的请求不重新读取活动指针，也不会在中途切换模型。

如果数据库尚无活动版本，升级后的首次运行继续使用环境线路。若已经存在活动数据库版本但该版本损坏、引用被破坏或无法解密，系统必须 fail closed，返回稳定的 503 错误并告警，不能静默改用未获管理员选择的线路。

## 6. 数据模型

### 6.1 `ai_connections`

每一行代表一个运行参数不可变的连接版本，主要字段包括：

- 版本 ID、逻辑系列 ID、版本号和前一版本 ID；
- 管理员可读名称；
- 规范化 Base URL；
- 可选兼容 User-Agent；
- API Key 密文、随机 IV、认证标签和密钥版本；
- 运行配置摘要 `config_digest`；
- 创建时间、归档时间、删除时间和密钥销毁时间。

编辑连接时新增同系列版本，不原地覆盖活动或历史运行参数。归档、删除和密钥销毁属于单向生命周期字段，可以受控更新。

连接生成新版本时，系统在同一事务中为该连接当前未删除的模型预设生成等价新版本并改为引用新连接版本。模型 ID、协议和价格保持不变，但组合运行摘要发生变化，因此需要重新测试；管理员不必手工重建所有模型。

### 6.2 `ai_model_presets`

每一行代表一个运行参数不可变的模型预设版本，主要字段包括：

- 版本 ID、逻辑系列 ID、版本号和前一版本 ID；
- 关联连接版本；
- 显示名称和真实模型 ID；
- `responses` 或 `chat_completions` 协议；
- 可选 reasoning effort；
- 最大输出 token；
- 可选输入、输出每百万 token 美元价格；
- 运行配置摘要、创建时间、归档时间和删除时间。

归档和删除仍是单向生命周期字段。价格未填写时成本状态为“未知”，不得沿用其他模型价格或记为假零。环境目标继续使用现有环境费率作为其价格快照；数据库模型预设使用自身费率。

### 6.3 `ai_route_revisions` 与 `ai_route_targets`

`ai_route_revisions` 保存不可变的完整路由版本、前一活动版本、创建时间、激活时间和状态。

`ai_route_targets` 保存：

- 路由版本 ID；
- 位置 `0..5`，其中 0 为主线路；
- 来源类型 `database` 或 `environment`；
- 数据库来源对应的模型预设版本 ID；
- 激活时的安全显示快照和配置摘要。

数据库约束保证每个路由有一至六个不重复目标，位置连续且只有一个主线路。

### 6.4 `ai_runtime_state`

单例行保存当前活动路由版本和递增锁版本。激活请求必须携带读取时的 `expectedActiveRevision`，用乐观并发控制阻止两个管理页面互相覆盖。

### 6.5 `ai_config_events`

记录以下事件：

- 创建、编辑、归档和删除连接或模型；
- 模型发现请求；
- 真实连接测试及稳定结果码；
- 路由激活、冲突、失败和回退；
- 密钥销毁和解密故障。

事件只保存版本 ID、安全摘要、状态、延迟和稳定错误码，不保存 API Key、密文、请求正文、Provider 原始响应或访客内容。审计事件保留 180 天后由 Worker 清理。

### 6.6 `interaction_provider_attempts`

现有对话记录继续把 `provider` 和 `model` 表示为最终成功目标，同时新增 `interaction_provider_attempts` 保存与 interaction turn 关联的安全尝试记录，至少包含：

- 路由版本和目标位置；
- 连接与模型预设版本 ID；
- 成功、失败、超时或停止状态；
- 首字节与总延迟；
- Provider 返回时可用的输入、输出 token；
- 使用该模型价格快照计算的成本，或明确的未知状态。

这样备用线路失败后产生的可见 usage 不会错误套用最终模型价格。尝试记录沿用 interaction turn 的 10 天保留边界。

## 7. 加密与密钥处理

- API Key 使用 AES-256-GCM 加密后入库。
- 每次写入生成独立的 96-bit 随机 IV。
- AAD 绑定连接版本 ID、逻辑系列 ID 和密钥版本，防止不同记录之间替换密文。
- 浏览器响应只包含 `hasApiKey: true/false`，不返回明文、密文、认证标签或 Key 尾号。
- 编辑已有连接时，Key 输入为空表示沿用旧 Key；填写新值才创建包含新密文的新版本。沿用旧 Key 仍需密码复验；若 Base URL 的 origin 发生变化，还必须明确确认旧 Key 将发送到新的 origin。
- 解密只发生在服务端运行配置解析和显式连接测试路径。
- Provider SDK 错误在进入应用日志前必须经过净化，不能输出请求头、Authorization 或完整响应正文。
- 删除历史配置时通过清空密文、IV 和认证标签完成密钥销毁；删除后的旧路由只能用于解释，不能重新激活。

若加密主密钥丢失，数据库中的 Key 不可恢复。恢复路径是管理员重新录入中转 Key、测试并激活新版本，不能降级读取明文副本。

## 8. 管理后台信息架构

### 8.1 共享后台外壳

- `/admin` 保留“对话复盘”；
- 新增 `/admin/api` 作为“API 配置”；
- 两个页面共享管理登录状态、会话有效期、顶部导航和退出操作；
- 现有 `AdminConsole.tsx` 不继续承载 API 管理状态，避免对话复盘与高风险配置互相耦合。

后台继续使用现有全局 token、深色工作台视觉和紧凑信息密度，不引入另一套视觉语言，不使用装饰性卡片堆叠。

### 8.2 API 配置首屏

首屏必须直接显示：

- 当前主模型和中转；
- 活动路由版本；
- 备用线路数量和顺序；
- 最近激活时间；
- 每条线路最近测试状态；
- 主操作“编辑路由”。

管理员进入页面后无需先打开详情即可判断全站当前使用哪条线路。

### 8.3 路由编辑器

- 固定支持一条主线路和最多五条备用线路；
- 支持拖动排序，也提供上移、下移和移除按钮；
- 键盘操作不依赖拖动；
- 修改先形成浏览器草稿，不自动生效；
- 页面底部显示稳定的“放弃更改 / 激活配置”操作条；
- 激活前展示新旧差异、测试门状态和密码复验；
- 过期或失效测试直接给出“重新测试”入口。

### 8.4 中转配置库

桌面使用紧凑列表加详情 inspector：

- 列表展示名称、Base URL 域名、模型数量、测试状态、归档状态和是否正在使用；
- 详情展示模型预设、运行参数、最近测试、版本历史和安全操作；
- 新增中转采用两步流程：连接信息，然后第一个模型预设；
- API Key 使用密码输入，提供显隐控制，但已保存 Key 永不回填；
- 获取模型列表是显式辅助操作，失败后仍允许手动输入模型 ID；
- 删除入口使用明确危险确认，不把不可逆操作放在普通主按钮位置。

390px 移动端先显示列表，详情、编辑和路由排序进入全屏层，沿用现有对话详情的返回模式。长 Base URL、长模型 ID、六条路由、加载、空状态、测试失败和会话过期不得造成横向溢出。

## 9. 管理 API

管理接口沿用现有 `/api/admin` 鉴权边界。URL 中的连接和模型 ID 均为稳定逻辑系列 ID；具体不可变版本只由服务端解析和返回只读摘要：

- `GET /api/admin/providers/runtime`：活动路由、只读环境线路和脱敏状态；
- `GET /api/admin/providers`：连接、模型和版本摘要；
- `POST /api/admin/providers`：创建连接及首个模型；
- `PATCH /api/admin/providers/[connectionId]`：创建连接新版本；
- `DELETE /api/admin/providers/[connectionId]`：执行受约束删除；
- `POST /api/admin/providers/[connectionId]/models`：创建模型预设；
- `PATCH /api/admin/providers/models/[modelId]`：创建模型新版本；
- `DELETE /api/admin/providers/models/[modelId]`：归档或删除模型；
- `POST /api/admin/providers/[connectionId]/discover`：显式调用兼容 `/models`；
- `POST /api/admin/providers/models/[modelId]/test`：显式真实测试；
- `POST /api/admin/providers/runtime/environment/[targetKey]/test`：显式测试只读环境主节点或 fallback，不允许借此修改环境配置；
- `POST /api/admin/providers/routes/activate`：原子激活或回退；
- `GET /api/admin/providers/events`：分页读取安全审计事件。

所有响应设置 `Cache-Control: no-store`。所有写操作和外部网络操作必须通过现有管理员 Session 与精确 Origin 校验。新增或替换 Key、任何 Base URL/origin 变化、模型发现、真实测试、激活、回退、归档和删除还必须重新验证管理员密码。沿用旧 Key 但改变 origin 时，接口还要求独立的显式确认字段，不能把空 Key 输入静默解释为可跨 origin 复用凭据。

请求体使用显式结构化字段和严格长度、枚举、数值边界校验。接口不得接受任意 Provider headers；首版只允许受控 User-Agent 覆盖，防止管理员界面变成通用 SSRF/HTTP 客户端。

## 10. Base URL 与 SSRF 防护

生产 Base URL 必须满足：

- 使用 HTTPS；
- 不包含用户名、密码、查询参数或 fragment；
- 允许规范化路径，例如 `/v1`；
- 主机名不能是 localhost、私网 IP、链路本地地址、组播地址或云元数据地址；
- 保存、测试和每次真实出站前都重新解析全部 A/AAAA 结果，并同时覆盖 IPv4-mapped IPv6；任一结果属于私有、保留、链路本地、组播或元数据地址时整体拒绝；
- 实际 TCP/TLS 连接必须绑定到本次已验证的公网 IP，同时保留原始 hostname 用于 Host 与 TLS SNI 校验，不能在检查后交给 SDK 再次自由解析；
- 首版禁止 Provider 重定向；发现、测试和 Chat 请求均以非重定向响应失败，避免跨 origin 或 DNS rebinding 绕过；
- 连接测试和 Chat Provider 都使用同一个受控出站策略。

本地自动测试只允许明确的 loopback Mock 注入，不通过生产配置白名单放宽上述规则。

## 11. 模型发现与真实连接测试

### 11.1 模型发现

模型发现由管理员主动触发兼容 `GET /models`：

- 不自动轮询；
- 不把发现成功视为 Chat 协议可用；
- 返回模型 ID 的安全子集；
- Provider 不支持时返回稳定错误，并保留手动填写入口。

### 11.2 真实测试

连接测试绑定完整运行摘要，包括连接版本、Key、Base URL、模型、协议、reasoning effort、User-Agent 和输出限制。数据库模型通过逻辑模型 ID 测试；只读环境目标通过稳定的 `primary`、`fallback-1`、`fallback-2` target key 测试，结果同样绑定环境配置摘要且不返回环境 Key。

测试调用：

- 不读取访客数据、对话历史、RAG、Embedding 或搜索；
- 使用固定的极短提示词；
- 限制最小合理输出；
- 走所选 `responses` 或 `chat_completions` 流式协议；
- 记录是否完成、延迟、usage 和净化后的稳定错误码；
- 单个管理员最多一个并发测试，每分钟最多三次。

保存配置不会触发测试。按钮和确认层必须明确说明真实测试可能产生极小费用。

### 11.3 测试门

- 新目标或运行摘要发生变化的目标，激活前必须具有最近 30 分钟内的成功测试；
- 当前活动路由中未变化的目标，在只排序或移除其他目标时无需重新测试；
- 任何 Key、Base URL、模型、协议、reasoning effort、User-Agent 或输出限制变化都会使旧测试失效；
- 环境主节点和现有 fallback 节点在首次迁移时可继承当前活动状态，但之后重新加入其他路由时也需要显式测试；
- 前一活动版本在切换后 30 分钟内允许直接回退，超过窗口则按新增目标规则重新测试。

## 12. 激活、并发与回退

激活在单个数据库事务中完成：

1. 验证管理员 Session、Origin 和密码复验；
2. 锁定 `ai_runtime_state`；
3. 比对 `expectedActiveRevision`；
4. 验证一至六个有序目标、删除状态和测试门；
5. 写入不可变 route revision 与 targets；
6. 原子更新活动指针；
7. 写入安全审计事件；
8. 提交后返回新活动版本。

并发版本不匹配时返回 `409 AI_CONFIG_CONFLICT`，不做部分写入。前端刷新当前活动版本并重新展示差异，不能自动覆盖。

回退不是修改旧版本，而是创建一个引用可运行旧目标的新活动版本。已销毁 Key 的历史线路不可回退。

## 13. Provider 路由与超时

- Provider Router 接受一至六个完全独立的目标配置；每个目标有自己的客户端、Base URL、Key、模型、协议和 reasoning effort。
- Embedding Provider 始终独立，不随 Chat fallback 切换。
- 所有目标共享现有生成并发门，不能把六条备用线路变成六倍并发额度。
- 整个 Provider 阶段使用一个全局截止时间，不再使用“单节点总超时 × 节点数”。首版把现有 `MORSE_PROVIDER_TOTAL_TIMEOUT_MS` 明确定义为整条路由共享的总截止时间。
- 每个目标使用 `MORSE_PROVIDER_FIRST_BYTE_TIMEOUT_MS` 作为首字节预算；首字节前发生认证、协议、网络、超时或无输出错误时，可尝试下一目标。
- 一旦向访客输出任何正文 delta，后续错误直接结束本次流，不切换目标拼接另一模型的回答。
- 用户停止、请求取消或全局截止时间到达后立即终止，不继续尝试备用线路。
- 当剩余全局时间不足以完成下一次最小尝试时直接结束，避免形式上遍历六条但没有可用执行时间。

具体超时数值继续由环境运行限制控制，不在首版管理 UI 中开放。默认总截止时间仍为 90 秒、单目标首字节预算仍为 20 秒；因此配置六条线路不承诺每次故障都遍历全部六条。

## 14. 日志、成本与可观测性

Provider 端口必须把每次尝试和最终胜出目标的安全元数据作为结构化事件返回，不能再由 `chat-service` 从进程启动时的静态配置猜测。interaction turn 进入 `completed`、`failed` 或 `stopped` 终态时，尝试记录与终态更新在同一个数据库事务中幂等持久化；同一次重放不得生成重复尝试。

完成回答后，interaction turn 必须记录真正成功的：

- 中转显示名与安全内部 ID；
- 模型 ID；
- 路由版本；
- 目标位置；
- 协议和延迟。

每次尝试的 usage 按其模型价格快照独立估算，再汇总为整次交互成本。Provider 未返回 usage 或价格未配置时，对应成本保持未知，不能记为零；汇总同时保存已知小计与完整性状态，只有所有尝试都可计价时才把总成本标记为完整。

日志和管理导出不得包含 Base URL 全路径中的敏感查询数据；本设计已禁止 Base URL 查询参数。日志不得包含 Key、Authorization、Provider 请求正文或原始错误响应。

## 15. 删除语义

删除分为三类：

### 15.1 从未使用

连接或模型从未进入活动/历史路由，也没有 interaction 引用时，管理员密码复验和危险确认后可以物理删除。删除单个模型只清理该模型系列及版本，不影响同一连接的 Key；删除整个连接时才一并清理其全部未使用模型版本和密钥材料。审计事件保留安全 ID 和删除结果。

### 15.2 有历史引用但当前未使用

不能删除解释历史所需的非敏感元数据。删除单个历史模型时只保留该模型的 tombstone，禁止它继续测试、编辑、入路由或回退；只要同一连接仍有其他可运行模型，就不得销毁共享连接 Key。

删除整个历史连接时，系统执行：

- 销毁 API Key 密文、IV 和认证标签；
- 标记连接及相关不可运行版本为 deleted；
- 从默认配置列表隐藏，在“已删除”筛选中保留名称、模型、版本、删除时间和历史引用；
- 禁止测试、编辑、加入路由或回退。

这属于不可恢复的密钥销毁，不是普通归档。

### 15.3 正在使用

活动主线路或备用线路不能删除。接口返回 `409 AI_CONFIG_IN_USE`，界面提供“编辑路由并移除”的下一步。新路由激活后才能再次执行删除。

环境应急线路不能删除、归档或编辑。

## 16. 稳定错误合同

至少提供以下公开错误码：

- `AI_CONFIG_UNAVAILABLE`：配置服务或主密钥不可用；
- `AI_CONFIG_INVALID`：字段、URL、协议或路由结构无效；
- `AI_CONFIG_CONFLICT`：活动版本已被其他管理操作更新；
- `AI_CONFIG_TEST_REQUIRED`：新增或变化目标缺少有效测试；
- `AI_CONFIG_TEST_FAILED`：真实测试未完成；
- `AI_CONFIG_IN_USE`：目标仍在活动路由中；
- `AI_CONFIG_HISTORY_RETAINED`：目标只能执行密钥销毁和历史保留；
- `AI_CONFIG_SECRET_UNAVAILABLE`：密钥缺失、版本不匹配或解密失败；
- `AI_CONFIG_TARGET_DELETED`：路由引用了已删除目标；
- `AI_CONFIG_RATE_LIMITED`：模型发现或真实测试过于频繁。

管理界面把错误码映射为可执行下一步，不展示内部堆栈或 Provider payload。

## 17. 迁移与兼容

数据库迁移只新增表、约束和必要的 interaction 尝试记录结构，不重写已有对话正文或知识数据。

新增迁移一旦应用即成为数据库兼容基线。当前 readiness 会校验完整 migration manifest，因此应用该迁移后不能回退到只认识旧 manifest 的二进制；应用回退只允许回到已经识别新 migration、但仍默认使用环境 Provider 的兼容基线，不执行破坏性 down migration。每个后续阶段都必须能在这条基线上关闭自己的新入口或恢复环境路由。

升级顺序：

1. 部署前生成并安全注入 Provider 加密主密钥和版本；
2. 执行新增迁移；
3. 启动仍使用当前环境 Provider；
4. readiness 验证环境线路和配置存储可用；
5. 管理员通过 `/admin/api` 新增、测试并激活首个数据库路由；
6. 激活成功后，新请求开始使用数据库版本；
7. 环境线路继续保留为只读应急目标。

旧的 `OPENAI_FALLBACK_1_*` 与 `OPENAI_FALLBACK_2_*` 在迁移期继续按当前环境路由解释。数据库路由激活后不与旧环境 fallback 自动拼接，避免出现后台看不到的隐藏线路。

## 18. 验证与验收

### 18.1 单元与合同测试

- AES-256-GCM 正常往返、随机 IV、AAD、篡改、错误密钥和错误版本全部覆盖；
- 所有管理响应、日志和错误不包含明文 Key、密文或 Authorization；
- Base URL、DNS 私网解析、重定向、协议、模型、价格和 token 边界覆盖；
- SSRF 覆盖全部 A/AAAA、IPv4-mapped IPv6、检查后 DNS rebinding、连接 IP 绑定、TLS hostname 校验和禁止重定向；
- 中转/模型版本化、测试摘要失效、路由数量与位置约束覆盖；
- 主线路首字节前失败后按顺序切换；输出正文后禁止切换；
- 全局超时、请求取消和用户停止不会继续尝试备用线路；
- 真正成功目标及逐次 usage/cost 归属正确；
- `completed`、`failed`、`stopped` 和终态重放均与尝试遥测同事务、幂等持久化；
- Embedding Provider 与 RAG 行为不受 Chat 路由变更影响。

### 18.2 数据库与 API 集成测试

- 迁移可在空库和已存在 S10/S11 数据的数据库上执行；
- 激活事务、乐观冲突、测试门、回退和三类删除路径覆盖；
- 管理员 Session、精确 Origin、密码复验、no-store 和测试限速覆盖；
- 物理删除不会破坏历史外键；密钥销毁后无法重新激活；
- 配置读写失败不产生半激活版本。
- 审计事件在 180 天边界由 Worker 清理，保留期内事件不被提前删除，清理失败可重试且不影响 Chat 请求。

### 18.3 Provider Mock

本地 Mock 同时覆盖 Responses、Chat Completions、`/models`、认证失败、协议不兼容、无输出、首字节超时、流中断和 usage 返回。

自动测试不得调用真实 Provider。真实中转只在管理员届时明确配置凭据并主动触发测试后验收；本设计确认不构成付费调用授权。

### 18.4 后台验收

在 1440×900 与 390×844 验证：

1. 登录并进入 API 配置；
2. 新增中转和第一个模型；
3. 模型发现失败时可手动填写；
4. 显式测试成功后加入路由；
5. 排列一条主线路和最多五条备用线路；
6. 查看差异、复验密码并激活；
7. 无重启情况下，新对话使用新活动版本；
8. 已开始流式回答不受活动版本切换影响；
9. 并发页面修改得到冲突提示；
10. 失效线路从路由移除后按历史情况物理删除或销毁 Key；
11. 30 分钟内可直接回退前一活动版本；
12. loading、empty、error、permission denied、long URL、long model ID 和六线路状态不溢出；
13. 浏览器控制台错误和浏览器直接发起的意外外部请求为零。

工程门槛：完整测试集通过、`npm run build` 通过、本地 dev 冒烟通过。上线前另行执行 Lighthouse Performance ≥ 90。
