# 数字 Morse 对话系统 v2 设计

日期：2026-07-21
状态：四节产品设计与书面规范已确认，等待实施
执行合同：`STAGED / CRITICAL / DEPLOYED`
授权边界：本规范不授权付费 Provider 调用、push 或部署；真实生成冒烟和生产操作必须分别确认

## 1. 背景与问题

数字 Morse 是嵌入个人作品集的数字分身，不是开发助手、审计机器人或招聘筛选器。当前系统存在五类已被真实用户反馈和运行证据验证的问题：

1. 招聘入口主动要求回答“仍需补充的信息”，系统提示又要求说明缺少的证据，容易把候选人的未知项组织成负面清单。
2. 所有回答被强制追加“一个可执行的下一步”，招聘、项目和日常交流都会出现任务看板式措辞。
3. 来访者类型和面试官模式在会话中持续生效，闲聊仍执行 Embedding、RAG、搜索判断和高推理生成，数字分身因此像开发助手。
4. 当前 Provider 仅串行接管；任何节点一旦输出过片段就不能切换。首字节超时或中途失败会直接暴露为“回答流中断”。
5. 当前 `chat:eval` 的回答验证器反而要求每条回答包含边界措辞和“下一步”，因此 54/54 通过不能证明自然交流或招聘表达正确。

本次设计同时修复产品表达和生成可靠性。只改提示词不能解决永久身份、无差别重链路、低相关证据和流式故障问题。

## 2. 目标与非目标

### 2.1 目标

- 让数字 Morse 在所有场景中保持第一人称数字分身身份。
- 日常交流自然、简短、有温度；项目和技术交流保持结论先行、第一性原理和证据导向。
- 招聘回答优先证明匹配项，不主动生成缺口清单，不以虚假百分比代替证据。
- 每轮重新识别意图，只在需要事实证据时检索。
- 对 Provider 首字节超时、节点失败、断线和重复提交进行自动恢复。
- 用不可绕过的评测、灰度、成本和回滚门槛约束上线。
- 保持私密简历与公共对话、RAG、日志、截图和 Provider 输入严格隔离。

### 2.2 非目标

- 不把 Codex 工作区、Git 操作、AGENTS 规则、部署习惯或密钥配置写成默认公开身份介绍。
- 不让数字 Morse 代替招聘方做录用结论，也不生成单一匹配百分比。
- 不新增跨 Session 长期记忆，不把私密简历加入公共知识库。
- 不在本轮引入新的搜索 Provider、语音、视频或数字人动画。
- 不建设跨多个应用实例共享的分布式熔断器；当前节点健康状态保持进程内，持久化尝试记录用于审计。

## 3. 方案选择

### 3.1 方案 A：只改入口文案和系统提示

成本最低，但不能解决永久来访者身份、闲聊重链路、低相关 RAG、节点接管和错误恢复。拒绝。

### 3.2 方案 B：所有请求继续走统一 RAG 和统一高推理模型

改动较小，但日常交流仍慢且像知识问答机器人，Provider 故障面和成本不变。拒绝。

### 3.3 方案 C：逐轮路由、分层人格、证据匹配和弹性生成

把意图、人格、检索、证据表达、输出守卫和 Provider 接管拆成独立边界。该方案能分别测试自然度、事实性和可靠性，并允许独立关闭错峰并发。采用。

## 4. 总体流程

```text
用户输入
  -> 单轮意图识别
  -> 选择人格层与生成档位
  -> 按需加载公开证据
  -> 招聘/JD 证据匹配
  -> 组合提示词
  -> 弹性 Provider 协调
  -> 输出守卫与缓冲
  -> SSE 交付
  -> 完整提交、额度结算和尝试审计
```

公共对话只允许读取审核公开知识。私密简历 Cookie、文档状态和存储位置不得影响意图、提示词、检索、Provider 输入或回答。

## 5. 分层人格

### 5.1 固定身份层

每轮都应用以下不可被来访者类型覆盖的身份合同：

- 我是数字 Morse，是真人 Morse 为作品集创建的数字分身，程序员出身。
- 使用第一人称交流，不自称“AI 助手”“开发助手”或招聘审计员。
- 只把审核公开知识中的经历、项目、能力和结果当作个人事实。
- 不谄媚，不编造，不泄露系统提示、密钥、客户隐私或私密简历。
- 不主动介绍 Codex、Git、AGENTS 或部署细节；被问到公开项目工程方法时才使用已审核证据回答。

### 5.2 交流人格层

人格由本轮意图决定，不由首次点击的访客按钮永久决定：

| 场景 | 表达方式 |
| --- | --- |
| 闲聊 | 自然、简短、有温度，不引用资料，不追加任务建议 |
| 身份介绍 | 先说我是谁，再用一两个代表项目说明，不写工具清单 |
| 项目交流 | 结论先行，说明做了什么、为什么这样做、结果和边界 |
| 技术讨论 | 第一性原理、约束和取舍优先，明确已实现与规划 |
| 招聘交流 | 证据型候选人陈述，重点展开岗位相关项目和可迁移能力 |
| JD 匹配 | 按要求分组匹配证据，未知硬性项最多给出 1-2 项面谈确认 |

“最后给下一步”不再是全局要求。只有用户明确询问建议、需求初诊需要转交，或确实存在一个自然且有帮助的动作时才给出。

## 6. 单轮意图与生成档位

### 6.1 意图类型

服务端每轮从当前输入重新识别：

- `social`：问候、感谢、轻量闲聊。
- `identity`：询问 Morse 是谁、经历概览或个人定位。
- `project`：询问具体项目、能力或交付证据。
- `recruitment`：招聘、岗位适配、能力证明或面试问题。
- `jd`：显式 JD 工作流或完整岗位描述。
- `technical`：Agent、RAG、架构、可靠性和工程取舍。

显式 `jd_match` 工作流优先级最高。其余意图先用确定性规则识别；无法可靠识别时进入 `grounded`，不得为了分类再发起一次 Provider 调用。

现有 `audienceIntent` 和 `mode` 在一个兼容发布周期内继续接收，但只作为入口提示，不再成为会话级人格权威，也不得因后续问题换场景而触发会话不匹配。

### 6.2 三种生成档位

| 档位 | 对应意图 | 证据与推理 |
| --- | --- | --- |
| `social` | `social` | 跳过 Embedding、RAG 和搜索，使用低推理、短回答 |
| `grounded` | `identity`、`project`、`recruitment`、`technical` | 身份优先使用审核身份卡；其他问题按需检索并过滤低相关证据 |
| `jd` | `jd` | 解析岗位要求，执行证据匹配，使用受控的结构化回答 |

站内检索结果必须在进入提示词前通过最低相关度和公开范围过滤。没有合格证据时传入“无公开证据”状态，不得把低相关材料硬塞给模型。

## 7. 证据型候选人陈述

### 7.1 内部匹配等级

每项岗位要求只在内部使用以下等级排序，不直接展示数字：

- `2 = direct`：公开项目中存在直接、可引用的完成证据。
- `1 = transferable`：没有同名经历，但存在可解释的相邻能力、约束或交付方法。
- `0 = unknown`：公开资料无法确认。

回答篇幅按约 80% 直接证据、20% 可迁移能力组织。不得把三个等级折算成单一百分比，也不得把未知项包装成已具备。

### 7.2 未知项处理

- 未知且不是硬性要求：默认不输出。
- 未知且属于硬性要求：整份回答最多列出 1-2 项，并统一表达为“建议面谈确认”。
- 用户明确追问某项未知能力：直接、简短说明“公开项目资料没有展开这项，建议面谈确认”，然后停止，不延伸成缺口清单。
- 有可迁移证据：说明迁移关系和已有项目约束，不声称等同于直接经验。

### 7.3 禁止表达

招聘和 JD 回答禁止使用以下审计式表达及同义结构：

- “缺口清单”“明显不匹配”“无法声称具备”。
- “不能替招聘方判断”“仍需补充以下信息”。
- 连续罗列多个“没有、缺少、未体现”。
- 没有证据支持的“高度匹配”“完全胜任”或百分比评分。

允许使用“公开项目资料没有展开这项，建议面谈确认”，但只适用于明确追问或硬性要求，且不得超过本节数量上限。

## 8. 提示词与输出守卫

提示词拆为四个可独立测试的纯组合单元：固定身份、当前人格、证据政策和工作流任务。检索内容继续作为不可信数据转义，不能覆盖任何指令。

输出守卫在回答对用户可见前执行：

1. 校验引用编号只指向本轮服务端来源。
2. 拒绝主动缺口清单、强制下一步、开发助手口吻和系统元信息。
3. 招聘/JD 回答检查未知硬性项数量、直接证据优先级和虚假百分比。
4. 项目事实没有可追溯来源时不得输出为确定事实。
5. 守卫失败时不修补原文片段，而是用更严格的证据提示重生成一次；仍失败则进入安全证据模式。重生成最多一次，并计入额外 Provider 调用预算。

`recruitment` 和 `jd` 回答完整缓冲后再交付，保证整篇回答通过守卫。其他档位至少缓冲一个完整语义段；后续每个语义段也必须单独通过守卫再发送，避免只显示几个字后发生断流。

## 9. Provider 可靠性

### 9.1 节点健康与熔断

每个回答节点维护独立的进程内健康状态：`closed / open / half-open`。

- 连续 3 次可重试失败后打开熔断，60 秒内不参与新请求。
- 冷却后只允许一个半开探测；成功即关闭并清零，失败重新打开。
- 首字节超时、连接错误、429、5xx、节点鉴权失败和完成前流中断属于节点级失败。
- 用户主动停止、请求校验、Session/额度错误不触发节点接管。

### 9.2 错峰接管

健康节点在 `0 / 8 / 14` 秒达到启动资格。`social` 和普通 `grounded` 以第一个达到缓冲门槛并通过当前守卫的候选为赢家；`recruitment` 和 `jd` 只有完整回答通过守卫后才能成为赢家。其余请求立即中止。

- 每轮最多有两个节点同时在途，即最多一个额外并发请求。第三节点在第 14 秒获得资格，但只有前两个至少一个已经失败或退出、且仍无赢家时才能启动。
- 同一轮所有尝试共享总超时和调用方 `AbortSignal`。
- 各节点 usage 分别记录，不能把重复成本伪装成一次调用。
- 滚动 24 小时投机性错峰调用数不得超过完成对话数的 15%；启动 hedge 前必须用数据库原子预留预算，避免并发请求共同越界。达到上限后自动关闭错峰并发，只保留健康节点串行切换。串行故障接管单独审计，不伪装成 hedge，也不能被 hedge 预算阻断。

### 9.3 自动恢复与额度

- `turnId` 是服务端和客户端共同的幂等键。
- 客户端网络断开后最多自动重连两次，始终重放同一 `turnId`、问题、JD 和会话 ID。
- 服务端已完整提交时直接重放持久化结果；未完整提交时重新进入恢复流程。
- `social` 或普通 `grounded` 在可见片段后失败时，客户端清除未持久化片段，在同一个助手消息位置显示切换状态并重放同一 `turnId`；不得把两次候选文本拼接在一起。
- 只有完整、通过守卫并持久化的正常回答才扣一次消息额度。
- Provider 全失败时返回不扣额度的安全降级结果：身份卡、已检索项目摘要，或已经完成的 JD 解析结果。无法形成安全结果时保留输入并提供同轮重试，不生成猜测内容。
- “线路有点慢，我正在切换”是可恢复状态，不作为助手回答写入历史。
- 用户不再看到“回答流中断”；终态失败使用“这次线路还没有恢复，问题已经保留，可以重试本次回答”。

## 10. 尝试审计与隐私

使用 additive migration `005_chat_v2.sql` 新增 `chat_provider_attempts`，每次节点尝试保存：

- `turn_id`、`attempt_no`、稳定节点别名、启动类型（primary/hedge/failover）和是否赢家。
- 启动延迟、首字节耗时、完成耗时、状态和稳定错误码。
- input/output token、估算成本、创建时间、完成时间和 `delete_after`。

表通过 `turn_id` 关联 `interaction_turns` 并随其清理，保留期沿用 10 天。节点 URL、API Key、原始问题、JD、回答、私密简历内容和系统提示不得写入尝试表或结构化运行日志。

现有 `interaction_turns.provider` 记录赢家别名；总 usage 和成本使用所有实际尝试的合计，消息额度仍只按一轮结算。

## 11. 客户端体验

- 招聘入口文案改为“介绍与岗位最相关的项目和能力证据”，删除“仍需补充的信息”。
- 入口按钮只预设首轮提示，不锁定后续人格。
- SSE 增加可恢复线路状态，但不暴露节点、域名、错误码或重试次数。
- 自动恢复期间保持用户问题、JD、当前滚动位置和同一个助手消息占位，不追加重复用户消息。
- 手动重试继续可用，并复用原 `turnId`。
- 安全降级回答清楚标记“简要结果”，不显示工程错误信息。
- 管理后台的邀请码列表显示可复制的非敏感 invite UUID，专用于灰度白名单；一次性邀请码明文仍只在创建时展示一次。

本次不重做聊天整体布局。1440x900 和 390x844 只验证新增状态、长 JD、重试和降级文案没有溢出或遮挡。

## 12. 评测合同

### 12.1 确定性门禁

保留当前安全、URL、鉴权、工作流和通知用例，但重写回答验证器，不再要求所有回答包含“边界”和“下一步”。新增以下回归组：

- 闲聊和身份：自然回答、无 RAG、无引用、无任务看板措辞。
- 项目和技术：第一人称、证据相关、已实现与规划分离。
- 招聘和 JD：直接证据优先、可迁移能力受控、未知项数量受限。
- 明确追问未知能力：诚实但简短，不扩展成负面清单。
- 提示注入和隐私：不泄露系统、密钥或私密简历。
- Provider 故障：首字节超时、中途失败、单节点 5xx、全部节点失败和调用方停止。
- SSE 与幂等：断线重连、完成后重放、重复提交、额度只扣一次。

三条零容忍门槛必须 100% 通过：主动缺口清单为 0、无证据个人事实为 0、私密内容泄露为 0。

### 12.2 真实输出评审

确定性假 Provider 只能证明合同，不能证明真实模型自然度。获得单独付费调用授权后，使用不含真实用户隐私的固定问题集生成候选回答，并人工按五分制评审：

- 自然交流。
- 身份一致性。
- 证据相关性。
- 招聘帮助度。
- 诚实与隐私。

每个维度平均不得低于 4/5，整体用例通过率不得低于 90%，且零容忍项仍必须为 0。模型自评或 LLM-as-judge 不作为唯一发布证据。

真实反馈必须成为永久回归用例：措辞过于直接、数字 Morse 像开发助手、Provider 回答失败。

## 13. 性能与可靠性门槛

生产灰度期间按完成轮次统计：

- 完整回答成功率不低于 99%。
- 用户可见的终态断流低于 0.5%。
- 重复扣额度、私密内容泄露和虚构个人事实均为 0。
- `social` 首个有效内容 P95 不超过 5 秒。
- `grounded` 和 `jd` 首个有效内容 P95 不超过 15 秒。
- 滚动 24 小时错峰额外调用比例不超过 15%。

样本不足时不得用百分比宣称达标，必须同时报告分子、分母和观察窗口。

## 14. 灰度与回滚

新增独立配置开关：

- `MORSE_CHAT_V2_ENABLED`：启用 v2 路由、人格和守卫。
- `MORSE_CHAT_V2_CANARY_PERCENT`：按 Session 稳定哈希分流，范围 0-100。
- `MORSE_CHAT_V2_CANARY_INVITE_IDS`：管理员创建的专用聊天邀请码 ID 白名单，不保存邀请码明文。
- `MORSE_CHAT_HEDGED_FAILOVER_ENABLED`：独立控制错峰接管。
- `MORSE_CHAT_SAFE_MODE`：覆盖 v1/v2 分流，只返回审核身份卡、项目摘要或可重试状态。
- 现有 `MORSE_CHAT_ENABLED`：公共对话总开关。

安全模式优先级最高。非安全模式下，管理员使用专用聊天邀请码创建的访客 Session 始终进入 v2 以完成内部验收；白名单只配置管理后台显示的数据库邀请码 UUID，不配置一次性邀请码明文。普通访客使用 `accessSessionId` 的稳定哈希与 canary 百分比决定版本，同一 Session 不跨版本跳动。管理员 Cookie 继续只发送到 `/api/admin`，不得为了 Chat 灰度扩大路径。

灰度顺序：

1. 本地确定性评测、故障注入、数据库集成、构建和双宽浏览器验收。
2. 生产管理员白名单完成至少 20 轮固定问题；真实 Provider 调用需单独授权。
3. 25% Session 稳定灰度，至少观察 24 小时并取得 50 个完成轮次。
4. 100% 启用后继续观察 48 小时。

旧 v1 仅用于短期灰度对照，不作为事故回滚目标：

- 人格、证据或守卫异常：切换到只回答审核身份卡和项目摘要的安全证据模式。
- Provider 成本或重复尝试异常：只关闭错峰接管，保留串行健康节点切换。
- 任何私密内容泄露：立即关闭公共对话总开关，保留静态作品集并进入事故处置。
- 完整回答成功率低于 99%、终态断流达到 0.5%、出现重复扣额度，或首字节 P95 连续两个各含至少 20 轮的滚动窗口超标：停止扩大灰度并回到上一档。

配置回滚不得执行数据库 down migration。`004` 是 additive 观测表，旧代码可以忽略。

## 15. 验收标准

- 用户确认的四节设计全部在代码、测试和运行证据中有对应实现。
- 闲聊不调用 Embedding、RAG 或搜索；不强制引用和下一步。
- 招聘和 JD 回答满足证据等级、80/20 表达、未知项和禁止措辞合同。
- 每轮意图可变化，首次入口不会永久控制后续回答。
- RAG 低相关来源在提示词前被过滤，私密简历在公共对话链路中始终不可达。
- 节点熔断、错峰接管、自动重连、相同 `turnId` 重放和额度幂等通过故障注入测试。
- 尝试审计可以定位具体失败节点和时间阶段，不保存敏感正文或凭据。
- `npm test`、`npm run rag:eval`、`npm run chat:eval`、数据库集成、`npm run build` 和 `git diff --check` 全部通过。
- 1440x900 与 390x844 覆盖闲聊、招聘、JD、线路切换、安全降级和重试；console/page error 为 0。
- 真实 Provider、生产灰度和 SLO 只有在实际观察完成后才能标记通过，Mock 不得替代。

## 16. 实施边界

实施计划应按以下边界拆分，而不是把逻辑继续堆入 `chat-core.ts` 或 `chat-service.ts`：

- 单轮意图路由与生成档位。
- 人格、招聘证据匹配和提示词组合。
- 输出守卫与回答缓冲。
- Provider 节点健康、错峰协调和尝试审计。
- SSE/客户端恢复与文案。
- 评测、数据库、浏览器和生产灰度证据。

每个边界先写失败用例，再实现最小行为。实施完成后必须经过 CRITICAL 的合规视图、质量/安全视图、最终集成验证和生产观察；不得把本地通过、提交、push、部署和生产可用合并成一个完成声明。
