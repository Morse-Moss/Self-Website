# 作品集项目展示精简设计

日期：2026-07-19
状态：对话设计已确认，待书面 Spec 复核
执行合同：`STAGED / STANDARD / LOCAL`

## 1. 产品判断

作品页服务招聘方和潜在客户。用户进入页面后，应在 10 至 20 秒内看懂每个项目是什么、解决了什么问题、摩斯完成了哪些技术实现，再决定是否展开了解。

页面采用“代表作品列表 + 原位展开详情”的内容型作品集模式。首层强调项目、能力和真实使用状态；工程细节逐层展开。设计过程、审计口径和内部证据元数据不进入公开叙事。

## 2. 本轮范围

本轮统一简化作品页标题、折叠卡片和展开详情结构，并完成“内容创作 Agent 系统”的公开文案与知识库事实同步。

其他项目沿用同一展示结构，但本轮只对现有事实做保守迁移，不新增卖点、不改写未经逐项确认的项目叙事。后续继续按“一次完善一个项目”的方式终审。

本设计取代以下旧公开展示约定：

- `WORK INDEX`、超大“作品”标题和作品页说明段落。
- 折叠卡片中的独立归属说明、未来方向长句、媒体说明段落和多个操作按钮。
- 展开详情中的“验证证据”“当前边界”以及采集时间、提交版本、运行方式、脱敏处理等媒体元数据。
- “问题 -> 我的角色 -> 关键判断 -> 真实结构 -> 验证证据 -> 当前边界”的旧章节顺序。

事实核验仍由内部资料、验证文档和测试负责，但不再把核验过程写给作品页访客。

## 3. 作品页首部

作品页首部只显示一行居中的标题：

> 代表作品

不显示副标题、英文眉题或解释性段落。标题使用页面级而非 Hero 级字号，让第一项作品在首屏内可见。

## 4. 折叠项目行

每个项目以一条横向作品记录展示，不使用嵌套卡片或装饰性面板。

桌面端左侧为项目主图，右侧依次为项目名称、最多两行简介、五个能力短词和一行状态。右下角仅保留展开图标。移动端改为单列，主图在上、内容在下，信息顺序不变。

交互与内容约束：

- 整行可点击，图标按钮保留原生按钮语义、`aria-expanded` 和明确的无障碍名称。
- 简介最多显示两行，长文本不会改变同列项目的基础节奏。
- 能力短词使用紧凑标签，数量固定为五个；不在首层展示完整技术栈。
- 项目状态用于传达开发归属与真实使用阶段，不混入“已验证”“证据充分”等内部口径。
- 主图为设计稿时，只允许在图片上使用低权重角标“界面设计稿 · 示例数据”，不再显示解释段落。
- “问数字摩斯”和外部链接移至展开内容末尾；折叠状态只承担快速判断与展开入口。
- 展开后不重复项目名称、简介、状态和主图，直接进入详情正文。

## 5. 展开详情

详情使用五个固定章节，技术栈放在最后：

1. 项目简介
2. 核心能力
3. 系统架构
4. 我的技术实现
5. 技术栈

各章节使用编号、标题、短段落或短列表，并由细分隔线组织，不做“卡片套卡片”。“系统架构”以一条主流程和五个模块名称展示；“我的技术实现”先说明业务协作与技术归属，再列个人实现范围，最后用一行标注未来方向。

若项目存在站内对话或 GitHub 入口，统一放在五个章节之后，避免打断阅读路径。

## 7. 内容创作 Agent：展开内容

### 01 项目简介

> 为企业电商内容团队开发的 AI 图片与视频创作系统，已部署至企业局域网并投入使用。系统集中管理商品素材、参考图、Prompt、Skill、生成任务和结果资产。
>
> 系统支持 GPT 式对话创作，统一适配 GPT Image 2、Seedance 2、Kling、Veo、Wan 等图像与视频模型，并提供异步任务追踪、失败恢复、版本管理和资产归档。

### 02 核心能力

- 对话式创作：多轮需求补充、参考图上传、方案确认、图片和视频生成。
- Prompt 与 Skill 管理：沉淀可复用提示词、创作流程和场景能力。
- 多模型适配：GPT Image 2、Seedance 2、Kling、Veo、Wan 等。
- 任务运行管理：排队、执行、恢复、完成、失败、重试。
- 素材与版本管理：关联商品素材、参考图、任务、版本和资产。
- 执行前校验：检查参考图、画幅、清晰度和模型输入组合。

### 03 系统架构

主流程：

> 商品素材 / 参考图 -> 创作对话 -> Agent 决策与上下文编译 -> 模型网关 -> OperationRun 异步任务 -> 结果回挂 -> 数字资产库

模块：`创作层` `Agent 层` `模型层` `任务层` `资产层`

### 04 我的技术实现

> 项目需求、产品方向和部分创意来自真实业务对接；我是项目唯一开发者，负责将这些需求完整实现为可运行系统。

- Agent 对话、决策路由、上下文编译。
- 图片、视频和编辑模型 Provider 适配。
- 前端、后端、数据模型和权限体系。
- OperationRun 生命周期、异步执行、重试和恢复。
- Prompt、Skill、素材、任务、版本和资产关联。
- Windows 交付包、单端口托管、企业局域网部署、健康检查和故障恢复。

未来方向：

> 基于历史任务、人工反馈和质量评估，优化 Prompt、Skill、模型选择与创作策略，逐步演进为可审核、可回退的自进化 Agent。

### 05 技术栈

- 前端：React 18、TypeScript、Vite、Ant Design 5、TanStack Query。
- 后端：FastAPI、SQLAlchemy、Pydantic、Alembic。
- 数据与任务：MySQL、Redis、ARQ。
- AI / Agent：LangGraph、确定性决策路由、Provider Adapter。
- 工程交付：单端口静态托管、Windows 交付包、健康检查。

## 8. 内容与数据合同

公开项目数据改为直接对应五段详情：

- `overview`：项目简介段落。
- `coreCapabilities`：可扫描的能力列表。
- `architecture`：主流程与模块列表。
- `implementation`：协作归属说明、个人实现列表和未来方向。
- `techStack`：现有分组技术栈，固定在最后渲染。

媒体公开字段只保留资源、尺寸、替代文本和短角标。采集时间、提交版本、运行方式、脱敏说明、验证证据和当前边界不再由公开组件渲染；需要保留的核验信息继续放在 `docs/verify/` 和内部事实资料中。

其他三个项目迁移到新字段时只重组当前已确认事实，不补写新能力。内容终审仍逐项目进行。

## 9. 知识库同步

数字摩斯的内容创作 Agent 知识必须与本页使用同一组已确认事实，至少能准确回答：

- 项目面向谁、已经在哪里投入使用。
- GPT 式创作流程及 Prompt、Skill、数字资产的作用。
- 已适配的图像与视频模型。
- Agent、模型网关、异步任务和资产回挂的系统链路。
- 业务协作输入与唯一开发者技术实现之间的归属关系。
- 自进化 Agent 是未来方向，不是当前已完成功能。

知识检索继续保留来源链接和事实约束，但默认回答应先讲项目价值与技术能力，不主动输出审计语言。公开页面删除“当前边界”章节不等于放宽事实边界。

## 10. 响应式与交互约束

- 1440 宽下，标题与至少第一项作品的核心内容进入首屏；项目图与文字保持明确的左右层级。
- 390 宽下改为单列，项目名、两行简介、五个标签、状态和展开按钮均不溢出、不遮挡。
- 展开与收起继续保持一次只打开一个项目、URL Hash 同步和滚动定位。
- `prefers-reduced-motion` 下取消展开持续动画，并保持状态切换可用。
- 焦点样式、键盘操作、图片替代文本和详情区域关联保持完整。
- 不增加外部运行时资源或新依赖，沿用现有 token、CSS Modules 和组件边界。

## 11. 验收标准

### 内容合同

- 作品页首部只有“代表作品”，不出现 `WORK INDEX` 和原说明段落。
- 折叠卡片不出现归属长段、未来方向、媒体说明、证据元数据或完整技术栈。
- 内容创作 Agent 的简介、五个能力短词、状态、五段详情和技术栈与本设计逐字一致。
- 详情章节严格按“项目简介、核心能力、系统架构、我的技术实现、技术栈”排列。
- 页面不出现“验证证据”“当前边界”“采集时间”“提交版本”“运行方式”“脱敏处理”。
- 内容创作 Agent 只使用已确认的黑金新版主页面图，不出现旧版界面。

### 工程与视觉

- 更新受影响的类型与合同测试，旧七章节断言被新五章节断言替代。
- `npm test` 与 `npm run build` 通过。
- 本地生产构建或开发服务完成 `/works` 冒烟，控制台零报错。
- 1440 与 390 双宽完成折叠、展开、切换项目、Hash 直达和收起验证。
- 两个视口均无横向溢出、文本遮挡、按钮位移或图片加载失败。
- 视觉复盘确认：第一眼能识别代表作品，首层信息可快速扫描，详情没有重复内容和审计式废话。

## 12. 非目标

- 本轮不修改内容创作 Agent 本体，不新增模型或 Provider。
- 本轮不实现自进化 Agent。
- 本轮不公开企业内网入口、客户数据、密钥、Provider 配置或真实业务素材。
- 本轮不重做作品集全站视觉，不修改首页和数字摩斯对话产品逻辑。
- 本轮不部署、不推送远端；远端同步仍需单独授权。
